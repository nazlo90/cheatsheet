Hoisting & var / let / const Warm-up

Hoisting rules & TDZ

▶

Hoisting is JS's behavior of moving declarations to the top of their scope before execution — so variables and functions can be referenced before their line in code.

JS engine does a two-pass scan: first collects declarations, then executes. Only declarations are hoisted — not initializations.

DeclarationHoisted?Initialized?Usable before?
var xYesYes (undefined)Yes → undefined
let xYesNo (TDZ)No – ReferenceError
const xYesNo (TDZ)No – ReferenceError
function declarationYesYes (full body)Yes ✓
classYesNo (TDZ)No – ReferenceError
console.log(x);  // undefined
var x = 5;

greet();          // works — function fully hoisted
function greet() { console.log("Hello"); }

sayHi();          // TypeError — var hoisted, value = undefined
var sayHi = function() {};

var vs let vs const + loop gotcha

▶
varletconst
ScopeFunctionBlockBlock
Re-declareYesNoNo
Re-assignYesYesNo (binding)
TDZNoYesYes
for (var i = 0; i < 3; i++) setTimeout(() => console.log(i), 0);
// 3 3 3 — shared binding
for (let i = 0; i < 3; i++) setTimeout(() => console.log(i), 0);
// 0 1 2 — new binding per iteration
const prevents re-assignment of binding, not mutation. const obj = {}; obj.x = 1 is valid.

Event Loop Core

Call Stack → Microtasks → Macrotasks

▶

The event loop is the mechanism that allows JS (single-threaded) to handle async operations — it continuously checks if the call stack is empty, then processes queued tasks.

  1. Call Stack — synchronous code runs here
  2. Microtask Queue — Promise.then/catch/finally, queueMicrotask(), MutationObserver
  3. Macrotask Queue — setTimeout, setInterval, UI events, I/O (one per loop tick)
After every single macrotask, the entire microtask queue is drained before the next macrotask.
console.log('1');
setTimeout(() => console.log('2'), 0);
Promise.resolve().then(() => console.log('3'));
console.log('4');
// Output: 1 → 4 → 3 → 2

Complex example + async/await

▶
setTimeout(() => {
  console.log('A');
  Promise.resolve().then(() => console.log('B'));
}, 0);
Promise.resolve()
  .then(() => { console.log('C'); return Promise.resolve(); })
  .then(() => console.log('D'));
console.log('E');
// E → C → D → A → B
async function main() {
  console.log('start');
  await Promise.resolve();
  console.log('after await'); // microtask
}
main();
console.log('sync');
// start → sync → after await

this, Context & Closures Core

5 rules for this + call/apply/bind

▶

The this keyword refers to the object that is currently executing the function — its value is determined by how the function is called, not where it's defined.

ContextValue of this
Global (non-strict)window / global
Global (strict mode)undefined
Object methodThe object before the dot
Constructor (new)The newly created object
Arrow functionLexical — from enclosing scope at definition time
call/apply/bindExplicitly set first argument
function greet(hi, punct) { return `${hi}, ${this.name}${punct}`; }
const p = { name: 'Bob' };
greet.call(p, 'Hi', '!');       // immediate, args spread
greet.apply(p, ['Hey', '.']);   // immediate, args array
const bound = greet.bind(p, 'Hello'); // returns new fn
bound('?');

Closures — definition & use cases

▶

A closure retains access to outer scope variables even after the outer function returns.

function makeCounter(init = 0) {
  let count = init;
  return { inc: () => ++count, dec: () => --count, val: () => count };
}
const c = makeCounter(10);
c.inc(); // 11
Use cases: module pattern, memoization, partial application, event handlers, data encapsulation, factory functions.

Higher-Order Functions

HOF, currying, memoization, composition

▶

A HOF either takes a function as argument or returns a function.

// Currying: Transform a multi-arg function into a sequence of single-arg functions.
const add = a => b => a + b;
const add5 = add(5);
add5(3); // 8

// Generic curry: Adapts any function — collects args until arity is satisfied, then calls.
const curry = fn => function curried(...args) {
  return args.length >= fn.length ? fn(...args) : (...more) => curried(...args, ...more);
};

// Memoize: Cache return values keyed by serialized args to skip repeated computation.
function memoize(fn) {
  const cache = new Map();
  return function(...args) {
    const key = JSON.stringify(args);
    if (cache.has(key)) return cache.get(key);
    const r = fn.apply(this, args);
    cache.set(key, r); return r;
  };
}

// Composition & pipe: compose = right-to-left (f(g(x))), pipe = left-to-right (g then f).
const compose = (...fns) => x => fns.reduceRight((v, f) => f(v), x);
const pipe    = (...fns) => x => fns.reduce((v, f) => f(v), x);

WeakMap & WeakSet — weak references

▶

WeakMap/WeakSet hold weak references — keys/items can be garbage-collected when there are no other references. Keys must be objects.

WeakMapWeakSet
Keys / ItemsObjects onlyObjects only
GC when no other refsYes ✓Yes ✓
IterableNo ✗No ✗

const cache = new WeakMap();

  if (cache.has(obj)) return cache.get(obj);
  const result = heavyWork(obj);
  cache.set(obj, result); // auto-GCd when obj goes out of scope
  return result;
}

Classes (all about)

Class syntax — full reference

▶

A class is syntactic sugar over prototype-based inheritance — a blueprint for creating objects with shared methods, private fields, and constructor logic. Under the hood, it's still prototype chains.

class Animal {
  // Class field (ES2022)
  #name;                          // private field
  species = 'unknown';           // public field
  static count = 0;              // static field

  constructor(name) {
    this.#name = name;
    Animal.count++;
  }

  // Getter / setter
  get name() { return this.#name; }
  set name(v) { this.#name = v; }

  speak() { return `${this.#name} makes noise`; }
  static getCount() { return Animal.count; }
}

class Dog extends Animal {
  #tricks = [];

  constructor(name) {
    super(name);           // must call super first
  }

  speak() {
    return super.speak() + ' — Woof!';
  }

  learn(trick) { this.#tricks.push(trick); }
}

// instanceof works through prototype chain
new Dog('Rex') instanceof Animal; // true

▶
// Abstract class simulation in TS
abstract class Shape {
  abstract area(): number;

// Mixin pattern (multiple inheritance simulation)
const Serializable = (Base) => class extends Base {
  serialize() { return JSON.stringify(this); }
};
const Validatable = (Base) => class extends Base {
  validate() { return Object.keys(this).length > 0; }
};
class User extends Serializable(Validatable(BaseModel)) {}

// Symbol.iterator — make object iterable
class Range {
  constructor(public from: number, public to: number) {}
  [Symbol.iterator]() {
    let cur = this.from;
    return { next: () => cur <= this.to
      ? { value: cur++, done: false }
      : { value: undefined, done: true } };
  }
}

Promises & async/await

Promise states, chaining, error handling

▶

A Promise is an object representing eventual completion or failure of an async operation. States: pending → fulfilled | rejected (immutable once settled).

// Create: Wrap an async operation — call resolve on success, reject on failure.
const p = new Promise((resolve, reject) => {
  setTimeout(() => resolve('done'), 1000);
});

// Chain — each .then returns a new Promise
p.then(v => v.toUpperCase())
 .then(v => console.log(v))
 .catch(err => console.error(err))
 .finally(() => console.log('done'));

// Static methods: Combinators for coordinating multiple Promises — differ in how they handle rejections.
Promise.all([p1, p2]);          // fail-fast: rejects if any rejects
Promise.allSettled([p1, p2]);   // waits all, returns {status, value/reason}[]
Promise.any([p1, p2]);          // first fulfilled
Promise.race([p1, p2]);         // first settled (fulfilled OR rejected)
Promise.resolve(val);
Promise.reject(err);

async/await — patterns & error handling

▶
// Basic: async/await makes async code look synchronous — always wrap in try/catch.
async function fetchUser(id) {
  try {
    const res = await fetch(`/api/users/${id}`);
    if (!res.ok) throw new Error(`HTTP ${res.status}`);
    return await res.json();
  } catch (err) {
    console.error(err);
    throw err;  // re-throw or return fallback
  }
}

// Parallel — don't await one by one!
const [users, posts] = await Promise.all([
  fetch('/api/users').then(r => r.json()),
  fetch('/api/posts').then(r => r.json())
]);

// Top-level await (ES2022, in modules)
const config = await loadConfig();
Never await inside a forEach — it doesn't work as expected. Use for...of for sequential or Promise.all(arr.map(...)) for parallel.

Generators & Iterators

Generator functions

▶

A generator function returns a Generator object. Execution pauses at each yield and resumes on .next().

function* counter(start = 0) {
  while (true) {
    const reset = yield start;  // can receive value via .next(val)
    start = reset ?? start + 1;
  }
}
const gen = counter(1);
gen.next();         // { value: 1, done: false }
gen.next();         // { value: 2, done: false }
gen.next(10);       // { value: 10, done: false } — reset

// Finite generator
function* range(start, end, step = 1) {
  for (let i = start; i < end; i += step) yield i;
}
[...range(0, 10, 2)]; // [0, 2, 4, 6, 8]

Custom iterators & async generators

▶
// Custom iterable object
const iterable = {
  [Symbol.iterator]() {
    let n = 0;
    return { next: () => n < 3
      ? { value: n++, done: false }
      : { value: undefined, done: true } };
  }
};
[...iterable]; // [0, 1, 2]

// Async generator — paginate API
async function* paginate(url) {
  let page = 1;
  while (true) {
    const data = await fetch(`${url}?page=${page++}`).then(r => r.json());
    if (!data.length) return;
    yield data;
  }
}
for await (const page of paginate('/api/items')) {
  process(page);
}

Big O & Data Structures

Array / Set / Map complexity

▶

Big O notation describes worst-case time or space complexity as input size (n) grows — ignoring constants. O(1) is instant; O(n) scales linearly; O(n²) is quadratic.

OperationArraySetMap
Access by index/keyO(1)—O(1)
Search (has/includes)O(n)O(1)O(1)
Insert end / addO(1) amortizedO(1)O(1)

Use Set for unique-value lookups (O(1) vs Array's O(n)). Algorithm complexity matters far more than loop style.

TypeScript Basics TS

Annotations, arrays, tuples, enums, unions

▶

TypeScript is a statically typed superset of JavaScript — it adds type annotations and catches type errors at compile time, then compiles to plain JS.

// Annotations
let n: number = 10;
const name: string = "Alice";
let active: boolean = true;
let val: unknown;   // must narrow before use
let x: any;         // escapes type system — avoid
function log(msg: string): void {}

// Arrays & tuples
let arr: number[] = [1, 2, 3];
let tuple: [string, number] = ["age", 30];
let readonly_arr: readonly number[] = [1, 2];

// Enums
enum Role { Admin = 'ADMIN', User = 'USER' }  // string enum
const enum Dir { Up, Down }  // const enum — inlined at compile time

// Union & intersection
type ID = string | number;
type AdminUser = Admin & User;

// Literal types
type Dir = "left" | "right" | "up" | "down";

// Functions
type Fn = (a: number, b: number) => number;
function greet(name: string, greeting?: string): string {
  return `${greeting ?? 'Hello'}, ${name}`;
}

unknown vs any

▶

Both accept any value, but unknown is the type-safe counterpart — you must narrow it before use, while any silently disables all type checking.

unknownany
Accepts any valueYes ✓Yes ✓
Assignable to other typesNo — must narrow firstYes — bypasses checks
Property/method accessError until narrowedAlways allowed
Type safetyPreserved ✓Disabled ✗
Use caseExternal data, API responsesMigration, quick escape hatch
let a: unknown = fetchData();
a.toFixed();            // ✗ Error — not narrowed yet
if (typeof a === 'number') a.toFixed(); // ✓ narrowed

let b: any = fetchData();
b.toFixed();            // ✓ no error — but runtime may explode
b.foo.bar.baz;          // ✓ TS is silent, runtime throws

// Prefer unknown for function params accepting arbitrary input
function parse(val: unknown): string {
  if (typeof val === 'string') return val;
  return String(val);
}

Auto-reconnect pattern + RxJS WebSocket

▶
// Reconnect with exponential backoff
class ReconnectingWS {
  private ws: WebSocket | null = null;
  private retries = 0;

  connect() {
    this.ws = new WebSocket(this.url);
    this.ws.onclose = () => {
      const delay = Math.min(1000 * 2 ** this.retries++, 30000);
      setTimeout(() => this.connect(), delay);
    };
  }
}

// RxJS webSocket
import { webSocket } from 'rxjs/webSocket';
const subject = webSocket('wss://api.example.com/ws');
subject.subscribe(msg => handle(msg));
subject.next({ type: 'subscribe', channel: 'prices' });
subject.pipe(
  retryWhen(errors => errors.pipe(delay(2000)))
).subscribe();
In Angular, always close WebSocket on ngOnDestroy. Use shareReplay(1) to share a single connection across multiple components.

Dependency Injection Angular

DI system — hierarchical injectors

▶

Angular has a hierarchical DI system. Each component has its own injector; if a dependency is not found, it walks up to the parent injector, then to the root injector.

Injector hierarchy: Platform → Root (AppModule) → Feature Module → Component → Element

// Singleton in root
@Injectable({ providedIn: 'root' })   // tree-shakeable
class AuthService {}

// Scoped to component (new instance per component)
@Component({ providers: [MyService] })
class MyComponent {}

// InjectionToken — for non-class values
const API_URL = new InjectionToken<string>('API_URL');
// providers: [{ provide: API_URL, useValue: 'https://...' }]
inject(API_URL); // functional injection (Angular 14+)

// @Optional — null if not found
constructor(@Optional() private logger?: LoggerService) {}

// @Self — only look in own injector
constructor(@Self() private svc: MyService) {}

// @SkipSelf — skip own, look in parent
constructor(@SkipSelf() private parent: MyService) {}

// @Host — stop at host element boundary
constructor(@Host() private svc: MyService) {}

// Factory provider
{ provide: Logger, useFactory: (http: HttpClient) => new RemoteLogger(http), deps: [HttpClient] }

// Multi-provider (interceptors, validators)

Change Detection Angular

Default vs OnPush + Zone.js

▶

Zone.js monkey-patches all async APIs and notifies Angular after async work. Default: checks whole tree. OnPush: skips subtree unless:

  • An @Input() reference changes (not mutation)
  • An event fires inside component
  • Observable emits via async pipe
  • markForCheck() / detectChanges() called
@Component({ changeDetection: ChangeDetectionStrategy.OnPush })
class CardComponent {
  @Input() data!: Data;   // must pass NEW reference

  constructor(private cd: ChangeDetectorRef) {}

  update() {
    this.cd.markForCheck();    // schedule check in next cycle
    // OR
    this.cd.detectChanges();   // immediate synchronous check
  }
  pauseCD()  { this.cd.detach(); }
  resumeCD() { this.cd.reattach(); }

Signals (Angular 16+, stable 19) New

signal / computed / effect

▶

A Signal is a reactive primitive that holds a value and notifies consumers automatically when it changes — enabling fine-grained change detection without Zone.js.

import { signal, computed, effect } from '@angular/core';

// Writable signal

count.set(5);
count.update(v => v + 1);

// Computed — auto-tracks dependencies

// Effect — runs side effects when deps change
effect(() => console.log('count changed:', count()));

// In template
// <p>{{ count() }}</p>  <!-- call as function -->

// Sync custom validator
function noSpacesValidator(control: AbstractControl): ValidationErrors | null {

    : null;

// Async validator (check unique email)
function emailTakenValidator(api: ApiService): AsyncValidatorFn {
  return (control) => of(control.value).pipe(
    debounceTime(300),
    switchMap(v => api.emailExists(v)),
    map(exists => exists ? { emailTaken: true } : null)
  );
}

// Cross-field validator (group level)
function passwordMatch(group: AbstractControl): ValidationErrors | null {
  const pwd = group.get('password')?.value;
  const confirm = group.get('confirmPassword')?.value;
  return pwd === confirm ? null : { mismatch: true };
}

NgRx Full Guide State

Core concepts + full example

▶

Store → Selector → Component → dispatch(Action) → Reducer/Effect → Store

// Actions: Plain objects describing what happened. They have a type and optional payload.
const loadUsers = createAction('[Users] Load');
const loadSuccess = createAction('[Users] Success', props<{ users: User[] }>());
const loadFailure = createAction('[Users] Failure', props<{ error: string }>());

// Reducers: Pure functions that take the current state and an action, and return a new state.
const reducer = createReducer(initialState,
  on(loadUsers, s => ({ ...s, loading: true })),
  on(loadSuccess, (s, { users }) => ({ ...s, users, loading: false })),
  on(loadFailure, (s, { error }) => ({ ...s, error, loading: false }))
);

// Selectors: Functions that select (extract) slices of state from the store.
const selectState  = createFeatureSelector<UsersState>('users');
const selectAll    = createSelector(selectState, s => s.users);
const selectById   = (id: string) => createSelector(selectAll, us => us.find(u => u.id === id));

// Effects: Handle side effects (like HTTP calls) by listening to actions and dispatching new actions asynchronously.
loadUsers$ = createEffect(() => this.actions$.pipe(
  ofType(loadUsers),
  switchMap(() => this.svc.getAll().pipe(
    map(users => loadSuccess({ users })),
    catchError(err => of(loadFailure({ error: err.message })))
  ))
));

NgRx tips & when NOT to use it

▶
  • Actions describe events, not commands — [Users Page] Load Requested
  • Reducers must be pure and immutable — never mutate state
  • Use selectors — decouple components from state shape + memoization
  • Use DevTools for time-travel debugging
Don't use NgRx for: simple feature-local state, single-component state, straightforward forms. Use BehaviorSubject services or Signals instead.

CI/CD best practices for frontend

▶
  • npm ci instead of npm install — uses exact lockfile, faster, reproducible
  • Cache node_modules by lockfile hash — skip install if unchanged
  • Run tests in headless Chrome — --browsers=ChromeHeadless
  • Fail fast — lint before tests, tests before build
  • Separate staging and production — deploy staging on merge to develop, prod on tag/release
  • Environment variables via secrets — never commit API keys
  • Nx affected — in monorepo, only test/build changed packages
  • Bundle size budget — configure in angular.json, fail CI if exceeded
// angular.json — bundle budget
"budgets": [
  { "type": "initial", "maximumWarning": "500kb", "maximumError": "1mb" },
  { "type": "anyComponentStyle", "maximumWarning": "2kb" }
]

Dockerfile for Angular app (multi-stage)

▶
# Stage 1 — build
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci                        # reproducible install
COPY . .
RUN npm run build -- --configuration=production

# Stage 2 — serve (nginx, ~23MB)
FROM nginx:alpine
COPY --from=builder /app/dist/my-app /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
# nginx.conf — handle Angular routing
server {
  listen 80;
  root /usr/share/nginx/html;
  index index.html;

  location / {
    try_files $uri $uri/ /index.html;   # SPA routing
  }

  location /api {
    proxy_pass http://backend:3000;     # proxy to backend
  }

  gzip on;
  gzip_types text/plain application/javascript text/css;
}

▶
TechTypeBest forLimit
SVGVector DOMCharts, diagrams, icons (<5k elements)Slow with many elements
Canvas 2DRaster JS APIGames, particles, large datasetsNot accessible per-element
