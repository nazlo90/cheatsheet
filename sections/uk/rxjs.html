Observable вищого порядку RxJS

switchMap vs mergeMap vs concatMap vs exhaustMap

▶

Оператори відображення вищого порядку підписуються на внутрішні Observable, що емітуються джерелом — вони відрізняються кількістю дозволених паралельних підписок та скасуванням попередніх.

ОператорСкасовує попередній?Паралельний?Найкраще для
switchMapТак ✓Ні (макс. 1)Пошук — важливий лише останній
mergeMapНіТак (необмежено)Паралельні незалежні запити
concatMapНіНі (черга)Послідовне, зі збереженням порядку
exhaustMapН/З (ігнорує нові)НіЛогін/відправка — без подвійного надсилання

Запобігання витокам пам'яті Ключове

3 патерни для запобігання витокам підписок

▶

Кожен subscribe(), що не прибирається, призводить до витоку пам'яті — коллбек зберігає посилання на компонент навіть після його знищення.

// ① async pipe — РЕКОМЕНДОВАНО для шаблонів (авто-відписка)
@Component({ template: `<li *ngFor="let u of users$ | async">{{ u.name }}</li>` })
class UserListComponent {
  users$ = this.http.get<User[]>('/api/users');
  // Немає subscribe(), не потребує очищення
}

// ② takeUntilDestroyed (Angular 16+) — РЕКОМЕНДОВАНО в тілі класу
class UserListComponent {
  private destroyRef = inject(DestroyRef);

  ngOnInit() {
    interval(1000).pipe(
      takeUntilDestroyed(this.destroyRef)
    ).subscribe(tick => this.update(tick));
    // автоматично завершується при знищенні компонента
  }
}

// ③ Ручна відписка — запасний варіант для складних випадків
class UserListComponent implements OnDestroy {
  private subs = new Subscription();

  ngOnInit() {
    this.subs.add(stream1$.subscribe(...));
    this.subs.add(stream2$.subscribe(...));
  }
  ngOnDestroy() { this.subs.unsubscribe(); }  // прибирає всі одразу
}

// Бонус — toSignal() також авто-відписується
users = toSignal(this.http.get<User[]>('/api/users'), { initialValue: [] });
// Немає subscribe, немає очищення — Angular управляє всередині
ПатернНайкраще дляВерсія Angular

Cold vs Hot + типи Subject

▶

Subject — це одночасно Observable і Observer; він транслює значення всім поточним підписникам. Cold Observable починає з чистого аркуша для кожного підписника; Hot Observable ділить одне виконання.

shareReplay — пастка refCount

▶

shareReplay мультикастить та відтворює останні N значень. Опція refCount контролює, чи зберігається підписка на джерело коли кількість підписників падає до нуля.

// ⚠ ВИТІК ПАМ'ЯТІ — refCount: false (дефолт до RxJS 6.6)
users$ = this.http.get('/api/users').pipe(
  shareReplay({ bufferSize: 1, refCount: false })
);
// Навіть після відписки всіх підписників, внутрішня підписка ЗАЛИШАЄТЬСЯ ВІДКРИТОЮ
// Якщо надано в 'root', витікає на весь час життя застосунку — прийнятно
// Якщо в компоненті, що знищується і відтворюється — нова підписка на джерело щоразу!

// ✓ БЕЗПЕЧНО — refCount: true (рекомендовано)
users$ = this.http.get('/api/users').pipe(
  shareReplay({ bufferSize: 1, refCount: true })
);
// Відписується від джерела, коли виходить останній підписник
// АЛЕ: наступний підписник ініціює новий HTTP-запит (не справжнє кешування)

// Правило: HTTP в root-сервісі → refCount: false нормально (синглтон)
// HTTP в сервісі рівня компонента → refCount: true для уникнення витоків
