CI/CD DevOps

Концепції CI/CD

▶

CI (Continuous Integration) — автоматичне збирання та тестування при кожному push/PR. Виявляє проблеми на ранніх етапах. CD (Continuous Delivery/Deployment) — автоматичне розгортання в середовища після проходження CI.

ЕтапЩо відбувається
ТригерPush до гілки / відкриття PR / створення тегу
Встановленняnpm ci — чиста, відтворювана установка
ЛінтингESLint, Stylelint — перевірка якості коду
ТестуванняЮніт-тести (Jasmine/Vitest), звіт покриття
Збиранняng build --prod — продакшн-бандл
E2EPlaywright на staging
РозгортанняPush до staging / prod (Kubernetes, S3 тощо)
СповіщенняSlack/email при успіху або помилці

GitHub Actions — приклад для Angular

▶
# .github/workflows/ci.yml
name: CI/CD Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - run: npm ci
      - run: npm run lint
      - run: npm run test -- --no-watch --browsers=ChromeHeadless --coverage
      - run: npm run build -- --configuration=production

      - name: Upload coverage
        uses: codecov/codecov-action@v3

  deploy-staging:
    needs: build-and-test
    if: github.ref == 'refs/heads/develop'
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to staging
        run: |
          aws s3 sync dist/app s3://my-app-staging
          aws cloudfront create-invalidation --distribution-id ${{ secrets.CF_ID }} --paths "/*"

GitLab CI — приклад для Angular

▶
# .gitlab-ci.yml
stages: [install, test, build, deploy]

variables:
  NODE_IMAGE: node:20-alpine

install:
  stage: install
  image: $NODE_IMAGE
  script: npm ci
  cache:
    key: $CI_COMMIT_REF_SLUG
    paths: [node_modules/]

test:
  stage: test
  image: $NODE_IMAGE
  script:
    - npm run lint
    - npm test -- --no-watch --browsers=ChromeHeadless
  coverage: '/Statements.*?(\d+(?:\.\d+)?)%/'

build:
  stage: build
  image: $NODE_IMAGE
  script: npm run build -- --configuration=production
  artifacts:
    paths: [dist/]
    expire_in: 1 week

deploy_staging:
  stage: deploy
  script: ./scripts/deploy-staging.sh
  environment: staging
  only: [develop]

Найкращі практики CI/CD для фронтенду

▶
  • npm ci замість npm install — використовує точний lockfile, швидше та відтворювано
  • Кешування node_modules за хешем lockfile — пропускати встановлення, якщо нічого не змінилось
  • Запускати тести у headless Chrome — --browsers=ChromeHeadless
  • Швидке завершення при помилці — лінтинг перед тестами, тести перед збиранням
  • Окремо staging та production — розгортати staging при злитті до develop, prod при тегу/релізі
  • Змінні середовища через secrets — ніколи не комітити API-ключі
  • Nx affected — у монорепо тестувати/збирати лише змінені пакети
  • Бюджет розміру бандлу — налаштовувати в angular.json, завершувати CI при перевищенні
// angular.json — бюджет бандлу
"budgets": [
  { "type": "initial", "maximumWarning": "500kb", "maximumError": "1mb" },
  { "type": "anyComponentStyle", "maximumWarning": "2kb" }
]
Моніторинг Must Know

Три стовпи: Метрики · Логи · Трейси

▶

Observability = здатність розуміти стан системи за її виходами. Три взаємодоповнюючі сигнали:

СтовпНа що відповідаєПриклади
МетрикиЧи здорова система? (числа в часі)LCP, частота помилок, використання пам'яті, затримка API
ЛогиЩо сталося? (події з контекстом)JS-помилки, дії користувача, відповіді API
ТрейсиКуди пішов час? (життєвий цикл запиту)Клік → виклик API → БД → таймлайн рендерингу
Тип моніторингуОписІнструменти
RUM (Real User Monitoring)Дані з браузерів реальних користувачів у продакшніSentry, DataDog RUM, New Relic Browser
СинтетичнийСкриптовані тести з контрольованих середовищ (без реальних користувачів)Playwright, Checkly, Lighthouse CI
Відстеження помилокПерехоплення, групування, сповіщення про JS-виняткиSentry, Bugsnag, Rollbar
APM (App Performance)Кореляція бекенду та фронтендуDataDog, New Relic, Dynatrace
RUM показує реальний досвід користувачів. Синтетичні тести працюють 24/7 до того, як проблеми торкнуться користувачів. Використовуйте обидва підходи.

Core Web Vitals — метрики, що мають значення

▶

Орієнтовані на користувача метрики продуктивності від Google. Впливають на позицію в пошуку та досвід користувача.

МетрикаВимірюєДобреПогано
LCP — Largest Contentful PaintЗавантаження — коли основний контент видимий< 2.5s> 4s
INP — Interaction to Next PaintРеактивність — UI реагує на введення< 200ms> 500ms
CLS — Cumulative Layout ShiftВізуальна стабільність — елементи не стрибають< 0.1> 0.25
TTFB — Time to First ByteШвидкість відповіді сервера< 800ms> 1.8s
FCP — First Contentful PaintПерший відрендерений піксель< 1.8s> 3s
// Вимірювання Core Web Vitals у коді
import { onLCP, onINP, onCLS, onFCP, onTTFB } from 'web-vitals';

onLCP(metric => sendToAnalytics({ name: metric.name, value: metric.value }));
onINP(metric => sendToAnalytics({ name: metric.name, value: metric.value }));
onCLS(metric => sendToAnalytics({ name: metric.name, value: metric.value }));

// Також доступно через PerformanceObserver (нативний API браузера)
new PerformanceObserver((list) => {
  list.getEntries().forEach(entry => console.log(entry));
}).observe({ type: 'largest-contentful-paint', buffered: true });
INP замінив FID як Core Web Vital у 2024 році. FID вимірював лише першу взаємодію; INP вимірює всі взаємодії протягом усього часу роботи зі сторінкою.

Стратегія моніторингу продуктивності та сповіщень

▶
Що моніторитиМетрика / СигналСповіщати коли
JS-помилкиЧастота помилок (помилки / перегляди сторінок)Частота зростає > baseline + 2σ
Завантаження сторінкиLCP p75 (75-й перцентиль)> 2.5s
API-викликиЧастота HTTP 4xx / 5xx, затримка p955xx > 1%, затримка > SLA
Шлях користувачаВідтік з воронки, rage-клікиЗавершення оформлення замовлення падає > 5%
Витоки пам'ятіРозмір JS heap протягом сесіїHeap монотонно зростає
// Власна мітка продуктивності — вимірювання конкретного сценарію користувача
performance.mark('checkout-start');
// ... виконати дії
performance.mark('checkout-end');
performance.measure('checkout-duration', 'checkout-start', 'checkout-end');

const [measure] = performance.getEntriesByName('checkout-duration');
sendMetric({ name: 'checkout.duration', value: measure.duration });

// Angular-специфічне: відстеження продуктивності CD за допомогою профайлера (лише dev)
// import { enableProdMode } from '@angular/core';
// Використовуйте вкладку "Profiler" Angular DevTools у Chrome
  • p50 проти p95 проти p99 — завжди моніторте перцентилі, а не середні значення. Повільний p99 впливає на 1% користувачів, але може сигналізувати про реальну проблему.
  • Втома від сповіщень — сповіщати про симптоми (вплив на користувача), а не причини. "Частота помилок зросла в 3 рази" > "CPU на рівні 80%".
  • Runbooks — кожне сповіщення має містити посилання на runbook: що це означає, що перевірити, як виправити.

Dockerfile для Angular-застосунку (багатоетапний)

▶
# Етап 1 — збирання
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci                        # відтворювана установка
COPY . .
RUN npm run build -- --configuration=production

# Етап 2 — роздача (nginx, ~23MB)
FROM nginx:alpine
COPY --from=builder /app/dist/my-app /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
# nginx.conf — обробка Angular-роутингу
server {
  listen 80;
  root /usr/share/nginx/html;
  index index.html;

  location / {
    try_files $uri $uri/ /index.html;   # SPA-роутинг
  }

  location /api {
    proxy_pass http://backend:3000;     # проксі до бекенду
  }

  gzip on;
  gzip_types text/plain application/javascript text/css;
}

Основні команди Docker

▶
# Збирання та запуск
docker build -t my-app:latest .
docker run -p 4200:80 my-app:latest
docker run -d --name my-app -p 80:80 my-app:latest

# docker-compose
docker-compose up -d          # запустити всі сервіси
docker-compose down           # зупинити та видалити контейнери
docker-compose logs -f frontend # стежити за логами
docker-compose exec frontend sh # відкрити shell у контейнері

# Образи
docker images
docker pull nginx:alpine
docker push my-registry/my-app:v1.0.0
docker rmi my-app:latest

# Контейнери
docker ps                     # запущені контейнери
docker ps -a                  # всі контейнери
docker logs -f container_id
docker stop container_id
docker rm container_id
Багатоетапні збирання — необхідність: тримайте фінальний образ малим (використовуйте nginx:alpine замість node). Ніколи не запускайте від root у продакшні (USER nginx). Використовуйте .dockerignore для виключення node_modules, dist, .git.
