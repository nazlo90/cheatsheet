Основи GraphQL Основне

Схема, типи та операції

▶

GraphQL — це мова запитів і середовище виконання для API. Клієнти запитують рівно ті дані, які їм потрібні — без надмірного або недостатнього отримання.

КонцепціяОпис
SchemaСтрого типізований контракт між клієнтом і сервером (SDL)
QueryОтримання даних лише для читання
MutationОперація запису (створення / оновлення / видалення)
SubscriptionОновлення в реальному часі через WebSocket
ResolverФункція, що повертає дані для поля
ScalarЛистові типи: String, Int, Float, Boolean, ID
# Schema Definition Language (SDL)
type User {
  id:        ID!
  name:      String!
  email:     String!
  posts:     [Post!]!
  createdAt: String
}

type Query {
  user(id: ID!):  User
  users:            [User!]!
}

type Mutation {
  createUser(input: CreateUserInput!): User!
  deleteUser(id: ID!):               Boolean!
}

input CreateUserInput {
  name:  String!
  email: String!
}
! означає not-nullable. [Post!]! означає not-nullable список not-nullable об'єктів Post — і список, і кожен елемент гарантовано існують.

Синтаксис запитів клієнта

▶
# Запит з вибором полів
query GetUser($id: ID!) {
  user(id: $id) {
    id
    name
    posts {
      title
      publishedAt
    }
  }
}

# Mutation
mutation CreateUser($input: CreateUserInput!) {
  createUser(input: $input) {
    id
    name
  }
}

# Fragments — набори полів для повторного використання
fragment UserFields on User {
  id
  name
  email
}

query { users { ...UserFields } }

# Aliases — двічі запитати одне поле з різними аргументами
query {
  alice: user(id: "1") { name }
  bob:   user(id: "2") { name }
}
Resolvers & Context Ключове

Анатомія resolver'а (code-first з NestJS)

▶
import { Resolver, Query, Mutation, Args, ID,
         ResolveField, Parent, Context } from '@nestjs/graphql';
import { UsersService } from './users.service';
import { User } from './models/user.model';
import { CreateUserInput } from './dto/create-user.input';

@Resolver(() => User)
export class UsersResolver {
  constructor(private usersService: UsersService) {}

  @Query(() => [User], { name: 'users' })
  findAll() { return this.usersService.findAll(); }

  @Query(() => User, { nullable: true })
  user(@Args('id', { type: () => ID }) id: string) {
    return this.usersService.findOne(id);
  }

  @Mutation(() => User)
  createUser(@Args('input') input: CreateUserInput) {
    return this.usersService.create(input);
  }

  // Field resolver — виконується для кожного повернутого User
  @ResolveField('posts', () => [Post])
  getPosts(@Parent() user: User) {
    return this.postsService.findByAuthor(user.id);
  }
}
Підхід code-first автоматично генерує SDL з TypeScript-декораторів. Підхід schema-first використовує написаний вручну SDL і прив'язує до нього resolver'и.

Моделі ObjectType та InputType

▶
import { ObjectType, Field, ID, InputType } from '@nestjs/graphql';
import { IsEmail, IsString } from 'class-validator';

@ObjectType()
export class User {
  @Field(() => ID)
  id: string;

  @Field()
  name: string;

  @Field({ nullable: true })
  bio?: string;

  @Field(() => [Post])
  posts: Post[];
}

@InputType()
export class CreateUserInput {
  @Field()
  @IsString()
  name: string;

  @Field()
  @IsEmail()
  email: string;
}
ValidationPipe від Nest працює також з GraphQL-аргументами — додайте його глобально через app.useGlobalPipes та використовуйте декоратори class-validator на класах @InputType().
DataLoader & Проблема N+1 Performance

Проблема N+1 та розв'язання через DataLoader

▶

Коли запит до списку вирішує вкладені поля, наївні resolver'и виконують один запит до БД на кожного батька — це проблема N+1. DataLoader групує та кешує їх в один запит на запит.

// Без DataLoader — N+1
// 1 запит для users, потім 1 запит на кожного user для posts → N+1
@ResolveField('posts', () => [Post])
getPosts(@Parent() user: User) {
  return this.db.query('SELECT * FROM posts WHERE author_id = $1', [user.id]);
}

// З DataLoader — 1 групований запит
import DataLoader from 'dataloader';

export function createPostsLoader(db) {
  return new DataLoader<string, Post[]>(async (authorIds) => {
    const posts = await db.query(
      'SELECT * FROM posts WHERE author_id = ANY($1)', [authorIds],
    );
    // Групуємо за author_id, щоб DataLoader відобразив 1 результат на ключ
    return authorIds.map(id => posts.filter(p => p.author_id === id));
  });
}

// Реєструємо per-request у контексті GraphQL
const server = new ApolloServer({
  context: ({ req }) => ({
    user:        req.user,
    postsLoader: createPostsLoader(db),  // новий екземпляр = свіжий кеш
  }),
});

// У resolver'і
@ResolveField('posts', () => [Post])
getPosts(@Parent() user: User, @Context() ctx) {
  return ctx.postsLoader.load(user.id);
}
Завжди створюйте новий екземпляр DataLoader на кожен запит. Спільне використання між запитами призводить до витоку даних між користувачами.
Subscriptions & Реальний час Real-time

Підписки з PubSub

▶
import { PubSub } from 'graphql-subscriptions';
import { Subscription, Resolver, Mutation, Args } from '@nestjs/graphql';

const pubSub = new PubSub();

@Resolver()
export class PostsResolver {
  @Mutation(() => Post)
  async createPost(@Args('input') input: CreatePostInput) {
    const post = await this.postsService.create(input);
    pubSub.publish('POST_CREATED', { postCreated: post });
    return post;
  }

  @Subscription(() => Post, {
    // Необов'язковий фільтр — надсилати лише підписникам відповідного authorId
    filter: (payload, variables) =>
      payload.postCreated.authorId === variables.authorId,
  })
  postCreated(@Args('authorId') authorId: string) {
    return pubSub.asyncIterator('POST_CREATED');
  }
}
# Запит підписки на клієнті
subscription {
  postCreated {
    id
    title
    author { name }
  }
}
Для продакшну замініть in-memory PubSub на graphql-redis-subscriptions (RedisPubSub), щоб події працювали між кількома екземплярами сервера.
Auth & Guards у GraphQL Безпека

GqlAuthGuard та декоратор CurrentUser

▶
// gql-auth.guard.ts — адаптує HTTP guard для контексту GraphQL
import { ExecutionContext } from '@nestjs/common';
import { GqlExecutionContext } from '@nestjs/graphql';
import { AuthGuard } from '@nestjs/passport';

export class GqlAuthGuard extends AuthGuard('jwt') {
  getRequest(context: ExecutionContext) {
    const ctx = GqlExecutionContext.create(context);
    return ctx.getContext().req;   // витягуємо HTTP req з контексту GQL
  }
}

// current-user.decorator.ts
import { createParamDecorator, ExecutionContext } from '@nestjs/common';

export const CurrentUser = createParamDecorator(
  (data: unknown, ctx: ExecutionContext) => {
    const gqlCtx = GqlExecutionContext.create(ctx);
    return gqlCtx.getContext().req.user;
  },
);

// Використання в resolver'і
@UseGuards(GqlAuthGuard)
@Query(() => User)
me(@CurrentUser() user: User) {
  return this.usersService.findOne(user.id);
}
Обробка помилок у GraphQL Resilience

Помилки GraphQL проти HTTP-помилок

▶

GraphQL завжди повертає 200 OK на рівні HTTP. Помилки передаються всередині масиву errors у тілі відповіді.

// Структура відповіді з помилкою GraphQL
{
  "data": { "user": null },
  "errors": [{
    "message": "User not found",
    "locations": [{ "line": 2, "column": 3 }],
    "path": ["user"],
    "extensions": { "code": "NOT_FOUND" }
  }]
}

// Кидаємо GraphQL-помилки, видимі користувачу, у resolver'ах
import { GraphQLError } from 'graphql';

@Query(() => User)
async user(@Args('id') id: string) {
  const found = await this.usersService.findOne(id);
  if (!found) throw new GraphQLError('User not found', {
    extensions: { code: 'NOT_FOUND' },
  });
  return found;
}

// NestJS HttpException автоматично перетворюється ApolloDriver — без додаткових налаштувань
Ніколи не виставляйте сирі стектрейси або внутрішні деталі помилок у полі extensions у продакшні. Використовуйте formatError у конфізі Apollo для очищення.

formatError та власне маскування помилок

▶
// app.module.ts — конфігурація ApolloDriver
@Module({
  imports: [
    GraphQLModule.forRoot<ApolloDriverConfig>({
      driver: ApolloDriver,
      autoSchemaFile: true,
      formatError: (error) => {
        const code = error.extensions?.code;
        // Показувати деталі лише для відомих помилок, видимих користувачу
        if (['NOT_FOUND', 'BAD_USER_INPUT', 'UNAUTHENTICATED'].includes(code)) {
          return error;
        }
        console.error(error);  // логувати для внутрішнього аналізу
        return new GraphQLError('Internal server error');
      },
    }),
  ],
})
Продуктивність GraphQL & Найкращі практики Production

Обмеження складності та глибини запитів

▶

Без обмежень клієнти можуть створювати глибоко вкладені або ресурсомісткі запити, які вичерпають ресурси сервера.

import { createComplexityPlugin } from 'graphql-query-complexity';

// Параметр plugins для ApolloDriver
plugins: [
  createComplexityPlugin({
    maximumComplexity: 100,
    estimators: [
      fieldExtensionsEstimator(),   // читає @complexity з resolver'а
      simpleEstimator({ defaultComplexity: 1 }),
    ],
    onComplete: (complexity) => {
      if (complexity > 100) throw new GraphQLError('Query too complex');
    },
  }),
]

// Обмеження глибини
import depthLimit from 'graphql-depth-limit';
validationRules: [depthLimit(5)]   // максимум 5 рівнів вкладеності
Встановлюйте обмеження складності та глибини в будь-якому публічному GraphQL API. Без них один спеціально сформований запит може покласти сервер.

  playground:    process.env.NODE_ENV !== 'production',
})

// Директива @cacheControl в SDL (schema-first)
type User @cacheControl(maxAge: 60) {
  id:   ID!
  name: String!
}
