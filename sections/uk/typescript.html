TypeScript Основи TS

Анотації, масиви, кортежі, enum, union

▶

TypeScript — це статично типізований надбудова над JavaScript: він додає анотації типів та виявляє помилки типів під час компіляції, після чого компілюється у звичайний JS.

// Анотації
let n: number = 10;
const name: string = "Alice";
let active: boolean = true;
let val: unknown;   // потрібно звузити перед використанням
let x: any;         // вимикає систему типів — уникайте
function log(msg: string): void {}

// Масиви та кортежі
let arr: number[] = [1, 2, 3];
let tuple: [string, number] = ["age", 30];
let readonly_arr: readonly number[] = [1, 2];

// Enum
enum Role { Admin = 'ADMIN', User = 'USER' }  // рядковий enum
const enum Dir { Up, Down }  // const enum — підставляється під час компіляції

// Union та intersection
type ID = string | number;
type AdminUser = Admin & User;

// Літеральні типи
type Dir = "left" | "right" | "up" | "down";

// Функції
type Fn = (a: number, b: number) => number;
function greet(name: string, greeting?: string): string {
  return `${greeting ?? 'Hello'}, ${name}`;
}

any vs never vs unknown

▶

Три спеціальних типи, що визначають крайнощі системи типів: any вимикає перевірку типів, unknown — безпечний top type, never — bottom type, значення якого не може існувати.

anyunknownnever
Приймає значення відБудь-якого типу ✓Будь-якого типу ✓Нічого — порожня множина ✗
Присвоюється доБудь-якого типу ✓ (обхід)Лише unknown / anyКожного типу ✓ (bottom)
Доступ до властивостейЗавжди дозволеноПомилка до звуженняПомилка (недосяжний код)
Безпека типівВимкнена ✗Збережена ✓Збережена ✓
ЗастосуванняМіграція, швидкий обхідЗовнішні дані, відповіді APIПеревірка вичерпності, недосяжний код, порожні union
// any — повністю вимикає перевірку типів
let b: any = fetchData();
b.toFixed();            // ✓ немає помилки — але під час виконання може кинути
b.foo.bar.baz;          // ✓ TS мовчить

// unknown — потрібно звузити перед використанням
let a: unknown = fetchData();
a.toFixed();            // ✗ Помилка — тип ще не звужено
if (typeof a === 'number') a.toFixed(); // ✓ звужено

// never — значення, яке не може існувати
function throwErr(msg: string): never {
  throw new Error(msg);  // ніколи не повертає значення
}

// never у перевірці вичерпності — TS видасть помилку, якщо кейс пропущено
type Shape = { kind: 'circle' } | { kind: 'rect' };
function area(s: Shape) {
  switch (s.kind) {
    case 'circle': return 1;
    case 'rect':   return 2;
    default: const _exhaustive: never = s; // ✗ помилка, якщо фігуру додано, але не оброблено
  }
}

// never у умовних типах — фільтрує елементи
type NonNullable<T> = T extends null | undefined ? never : T;
type OnlyStrings<T> = T extends string ? T : never;
type R = OnlyStrings<string | number | boolean>;  // string

// Використовуйте unknown для параметрів функцій із довільним введенням
function parse(val: unknown): string {
  if (typeof val === 'string') return val;
  return String(val);
}
Правило: використовуйте unknown, коли тип невідомий (зовнішні дані) — це змушує вас звузити тип. Використовуйте never, щоб позначити неможливі гілки та фільтрувати елементи union. Уникайте any, крім крайнього випадку при міграції.

Структурна vs Номінальна типізація Співбесіда

Duck typing — форма важливіша за назву

▶

TypeScript використовує структурну типізацію: сумісність базується на формі типу (його властивостях), а не на назві або оголошенні. Два типи з однаковою структурою є взаємозамінними, навіть якщо не пов'язані між собою.

// Структурна типізація — однакова форма = сумісні
interface Point { x: number; y: number; }
class Vector { x = 0; y = 0; }

const p: Point = new Vector();  // ✓ — однакова форма

// Перевірка зайвих властивостей — лише для ЛІТЕРАЛІВ об'єктів
const p2: Point = { x: 1, y: 2, z: 3 };  // ✗ Помилка — зайва властивість z
const obj = { x: 1, y: 2, z: 3 };
const p3: Point = obj;               // ✓ — не літерал, перевірка не застосовується

// Вплив на Angular — структурна типізація означає: два сервіси
// з однаковим публічним API взаємозамінні на рівні типів:
class RealAuthService  { isLoggedIn() { return true; } }
class MockAuthService  { isLoggedIn() { return false; } }
// TestBed може замінити їх без спільного interface — структурний збіг достатній

// Симуляція номінальної типізації — branded types запобігають випадковому змішуванню
type UserId  = string & { readonly _brand: 'UserId'  };
type OrderId = string & { readonly _brand: 'OrderId' };

function asUserId(id: string): UserId { return id as UserId; }

function getUser(id: UserId) { /* ... */ }
const orderId = asOrderId('abc');
getUser(orderId);  // ✗ Помилка — OrderId не присвоюється до UserId
// Без branded types: обидва є string → тихо приймається

Використовуйте branded types, коли важлива доменна коректність (ID, валюти, одиниці). Freely використовуйте структурну типізацію для моків і DTO — це перевага, а не недолік.

Generics, mapped, conditional, infer

▶
// Generic з обмеженням
function getProperty<T, K extends keyof T>(obj: T, key: K): T[K] {
  return obj[key];
}

// Mapped type
type Nullable<T> = { [K in keyof T]: T[K] | null };
type Mutable<T> = { -readonly [K in keyof T]: T[K] };   // прибрати readonly
type Optional<T> = { [K in keyof T]?: T[K] };           // додати ?

// Conditional + infer
type UnwrapPromise<T> = T extends Promise<infer R> ? R : T;
type IsArray<T> = T extends any[] ? 'yes' : 'no';

// Template literal types
type EventName<T extends string> = `on${Capitalize<T>}`;

// Discriminated union + звуження
type Shape = { kind: 'circle'; r: number } | { kind: 'rect'; w: number; h: number };
function area(s: Shape): number {
  switch (s.kind) {
    case 'circle': return Math.PI * s.r ** 2;
    case 'rect':   return s.w * s.h;
  }
}

// Deep partial (рекурсивний)
type DeepPartial<T> = T extends object
  ? { [K in keyof T]?: DeepPartial<T[K]> } : T;

Типобезпечний API клієнт — mapped + conditional types на практиці

▶

Поєднуйте mapped types, conditional types та template literals для побудови HTTP клієнта, де параметри та відповідь кожного endpoint повністю типізовані під час компіляції.

// 1. Визначте контракт API як map типів
interface ApiRoutes {
  '/users':        { method: 'GET';  params: { page: number };   response: User[]   };
  '/users/:id':    { method: 'GET';  params: { id: string };      response: User     };
  '/users/create': { method: 'POST'; params: { body: NewUser };   response: User     };
}

// 2. Виведіть допоміжні типи через conditional types
type GetRoutes  = { [K in keyof ApiRoutes]: ApiRoutes[K]['method'] extends 'GET'  ? K : never }[keyof ApiRoutes];
type PostRoutes = { [K in keyof ApiRoutes]: ApiRoutes[K]['method'] extends 'POST' ? K : never }[keyof ApiRoutes];

type RouteParams<T extends keyof ApiRoutes>   = ApiRoutes[T]['params'];
type RouteResponse<T extends keyof ApiRoutes> = ApiRoutes[T]['response'];

// 3. Типізована функція клієнта
function apiGet<T extends GetRoutes>(
  url: T,
  params: RouteParams<T>
): Observable<RouteResponse<T>> {
  return http.get<RouteResponse<T>>(url, { params: params as any });
}

// 4. Використання — повністю типізовано, неправильні аргументи = помилка компіляції
apiGet('/users', { page: 1 });         // ✓ відповідь: User[]
apiGet('/users/:id', { id: 'abc' });  // ✓ відповідь: User
apiGet('/users', { id: 'abc' });      // ✗ Помилка — невірні params для /users

// Template literal type — автоматичне генерування назв подій
type Endpoint = keyof ApiRoutes;
type LoadAction<T extends string> = `load${Capitalize<T>}`;
type UserActions = LoadAction<'users' | 'orders'>;  // 'loadUsers' | 'loadOrders'
Цей патерн використовується у tRPC та типізованих обгортках fetch — тип клієнта повністю виводиться з map маршрутів, тому додавання нового endpoint в одному місці автоматично оновлює весь клієнт.
Abstract Classes vs Interfaces TSСпівбесіда

Основні відмінності з першого погляду

▶

Abstract classes можуть містити реалізацію (методи, поля, конструктори) і задавати контракт — вони знаходяться між конкретним класом та interface. Interfaces — це чисті контракти форми: нульовий runtime-слід, видаляються під час компіляції.

МожливістьAbstract classInterface
Існування під час виконанняТак — компілюється у JS класНі — видаляється під час компіляції
Конкретні методи / поляТак ✓Ні ✗ (лише сигнатури)
Логіка конструктораТак ✓Ні ✗
Множинне успадкуванняНі — одиночний extendsТак — множинний implements
Злиття оголошеньНі ✗Так ✓
Використання з instanceofТак ✓Ні ✗
Токен Dependency InjectionТак ✓ (Angular/NestJS)Ні — не є runtime значенням
Правило за замовчуванням: надавайте перевагу interface для чистих контрактів форми. Використовуйте abstract class, коли потрібна спільна реалізація, конструктор або runtime токен для DI.

Abstract class — спільна реалізація + контракт

▶
// Abstract class — не може бути створений безпосередньо
abstract class Animal {
  // Конкретне поле + конструктор — спільні для всіх підкласів
  constructor(protected readonly name: string) {}

  // Конкретний метод — успадковується як є
  describe(): string {
    return `I am ${this.name}`;
  }

  // Abstract метод — підкласи ЗОБОВ'ЯЗАНІ реалізувати
  abstract makeSound(): string;
}

class Dog extends Animal {
  makeSound() { return 'Woof'; }
}

const d = new Dog('Rex');
d.describe();   // 'I am Rex' — успадковано від Animal
d.makeSound();  // 'Woof'

// new Animal('x') — ✗ Помилка: не можна створити екземпляр abstract класу
d instanceof Animal; // ✓ true — runtime перевірка працює

Interface — чиста форма, нульові витрати під час виконання

▶
// Interface — лише на рівні типів, видаляється при компіляції
interface Serializable {
  serialize(): string;
  deserialize(raw: string): void;
}

// Клас може реалізовувати декілька interfaces
interface Loggable { log(): void; }

class UserRecord implements Serializable, Loggable {
  serialize()          { return JSON.stringify(this); }
  deserialize(raw)     { Object.assign(this, JSON.parse(raw)); }
  log()                { console.log(this); }
}

// Злиття оголошень — додати поля до існуючого interface
interface Serializable { version: number; }  // ✓ злито

// instanceof з interface — ✗ не працює (немає runtime токена)
// ur instanceof Serializable  — помилка компіляції

// Замість цього використовуйте type guard:
function isSerializable(v: unknown): v is Serializable {
  return typeof (v as any)?.serialize === 'function';
}

TypeScript Mixins TS

Що таке mixins і навіщо їх використовувати

▶

Mixin — це функція, яка приймає базовий клас і повертає розширений клас. Це патерн, що додає поведінку до класу без успадкування. Вирішує обмеження одиночного успадкування у TypeScript/JavaScript.

УспадкуванняMixin
Декілька базНі — розширює один класТак — складати декілька mixins
Runtime витратиЛанцюжок прототипівОднакові — ланцюжок прототипів
Tree-shakeableНіТак — включати лише потрібне
Альтернатива в AngularAbstract базовий компонентhostDirectives API (Angular 15+)
// Допоміжний тип для mixin
type Constructor<T = {}> = new (...args: any[]) => T;

// Mixin — додає відстеження часових міток
function Timestamped<TBase extends Constructor>(Base: TBase) {
  return class extends Base {
    createdAt = new Date();
    updatedAt = new Date();
    touch() { this.updatedAt = new Date(); }
  };
}

// Mixin — додає поведінку м'якого видалення
function SoftDeletable<TBase extends Constructor>(Base: TBase) {
  return class extends Base {
    deletedAt: Date | null = null;
    softDelete() { this.deletedAt = new Date(); }
    get isDeleted() { return this.deletedAt !== null; }
  };
}

// Складання декількох mixins на базовому класі
class Entity {
  constructor(public id: number) {}
}

class User extends SoftDeletable(Timestamped(Entity)) {
  constructor(id: number, public name: string) {
    super(id);
  }
}

const u = new User(1, 'Alice');
u.touch();          // з Timestamped
u.softDelete();     // з SoftDeletable
u.isDeleted;        // true
u.createdAt;        // з Timestamped
Тримайте mixins якомога без стану — вони повинні додавати поведінку, а не зберігати незалежний стан. В Angular надавайте перевагу hostDirectives для компонентної композиції поведінки; використовуйте TS mixins для звичайних сервісів та утилітних класів.

Обмежені mixins + злиття interfaces

▶
// Обмежена база — mixin вимагає конкретних членів у базовому класі
interface HasName { name: string; }

function Greetable<TBase extends Constructor<HasName>>(Base: TBase) {
  return class extends Base {
    greet() { return `Hello, ${this.name}!`; }
  };
}

// ✗ Помилка — база повинна мати .name
// class BadBase {} class Bad extends Greetable(BadBase) {}

// ✓ Працює — Person має .name
class Person { constructor(public name: string) {} }
class FriendlyPerson extends Greetable(Person) {}
new FriendlyPerson('Bob').greet(); // 'Hello, Bob!'

// Злиття interface для типу повернення mixin — покращує автодоповнення в IDE
interface User extends ReturnType<typeof Timestamped>,
                         ReturnType<typeof SoftDeletable> {}
TypeScript Decorators TS

Що таке decorators та їх типи

▶

Decorator — це спеціальна функція, яка прикріплює метадані або модифікує клас/властивість/метод під час оголошення. TypeScript підтримує experimental decorators (stage 2, використовуються Angular) та нові TC39 stage 3 decorators (TypeScript 5+).

Увімкніть через "experimentalDecorators": true у tsconfig (для Angular/NestJS). TC39 decorators працюють без цього прапора у TS 5+.

Тип decoratorЗастосовується доАргументи
ClassОголошення класу(target: Function)
MethodМетод класу(target, key, descriptor)
PropertyВластивість класу(target, key)
Accessorget/set(target, key, descriptor)
ParameterПараметр конструктора / методу(target, key, paramIndex)
// 1. Class decorator — додає метадані / обгортає клас
function Sealed(target: Function) {
  Object.seal(target);
  Object.seal(target.prototype);
}

@Sealed
class Config { apiUrl = 'https://api.example.com'; }

// 2. Method decorator — обгортає метод (напр. логування, вимірювання часу)
function Log(target: any, key: string, desc: PropertyDescriptor) {
  const orig = desc.value;
  desc.value = function(...args: any[]) {
    console.log(`${key} called with`, args);
    return orig.apply(this, args);
  };
}

class UserService {
  @Log
  findById(id: number) { /* ... */ }
}

// 3. Property decorator — позначити обов'язкові поля
function Required(target: any, key: string) {
  let val = target[key];
  Object.defineProperty(target, key, {
    get: () => val,
    set: (v) => { if (!v) throw new Error(`${key} is required`); val = v; }
  });
}

// 4. Parameter decorator — використовується у DI фреймворках (Angular, NestJS)
function Inject(token: any) {
  return (target: any, _: string | undefined, index: number) => {
    Reflect.defineMetadata('inject', token, target, String(index));
  };
}

// 5. Decorator factory — decorator з аргументами
function MinLength(min: number) {
  return function(target: any, key: string) {
    let val: string;
    Object.defineProperty(target, key, {
      set: (v: string) => {
        if (v.length < min) throw new Error(`${key} min length ${min}`);
        val = v;
      },
      get: () => val
    });
  };
}

Порядок виконання decorators + використання в Angular

▶
// Порядок виконання (знизу вгору для вкладених decorators)
@A   // обчислюється 1-м, застосовується 2-м
@B   // обчислюється 2-м, застосовується 1-м
class Foo {}
// Порядок: обчислити A → обчислити B → застосувати B → застосувати A

// Angular decorators — це class/property decorators під капотом
@Component({ selector: 'app-root', template: '...' })
class AppComponent {
  @Input() title = '';      // property decorator
  @Output() saved = new EventEmitter(); // property decorator
  @HostListener('click')    // method decorator
  onClick() {}
}

// NestJS decorators — той самий механізм
@Controller('users')
class UsersController {
  @Get('/:id')
  getUser(@Param('id') id: string) {}  // parameter decorator
}

// TC39 stage 3 (TS 5+, experimentalDecorators не потрібен)
function logged(target: any, ctx: ClassMethodDecoratorContext) {
  return function(...args: any[]) {
    console.log(`${String(ctx.name)} called`);
    return target.apply(this, args);
  };
}
class Service {
  @logged
  fetch() { /* ... */ }
}
Angular використовує reflect-metadata для зберігання інформації про типи під час проектування для DI. Decorator зберігає токен типу; інжектор зчитує його під час виконання. Завжди вмикайте "emitDecoratorMetadata": true разом з "experimentalDecorators" у Angular проектах.
