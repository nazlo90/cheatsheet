Архітектура WebRTC Основне

Огляд WebRTC та процес з'єднання

▶

WebRTC (Web Real-Time Communication) забезпечує peer-to-peer передачу аудіо, відео та даних безпосередньо в браузері без плагінів. Використовує три основні API: RTCPeerConnection, RTCDataChannel та getUserMedia.

Процес з'єднання (Offer/Answer):

  1. Ініціатор створює Offer (SDP) через createOffer()
  2. Ініціатор встановлює його як локальний опис, надсилає SDP одержувачу через сигналізацію
  3. Одержувач встановлює його як віддалений опис, створює Answer
  4. Одержувач встановлює відповідь як локальний опис, надсилає назад через сигналізацію
  5. Обидві сторони обмінюються ICE-кандидатами через сигналізацію
  6. Перевірки ICE-з'єднання завершено — P2P медіа передається
// Сторона ініціатора
const pc = new RTCPeerConnection(iceConfig);

// Додаємо локальні треки перед створенням offer
const stream = await navigator.mediaDevices.getUserMedia({ audio: true, video: true });
stream.getTracks().forEach(track => pc.addTrack(track, stream));

const offer = await pc.createOffer();
await pc.setLocalDescription(offer);
signalingChannel.send({ type: 'offer', sdp: offer.sdp });

// Після збору ICE-кандидата
pc.onicecandidate = ({ candidate }) => {
  if (candidate) signalingChannel.send({ type: 'ice', candidate });
};

// Отримуємо віддалені треки
pc.ontrack = ({ streams }) => { videoEl.srcObject = streams[0]; };
// Сторона одержувача
const pc = new RTCPeerConnection(iceConfig);
await pc.setRemoteDescription(new RTCSessionDescription(offer));
const answer = await pc.createAnswer();
await pc.setLocalDescription(answer);
signalingChannel.send({ type: 'answer', sdp: answer.sdp });

// Додаємо отримані ICE-кандидати
await pc.addIceCandidate(new RTCIceCandidate(candidate));

ICE, STUN та TURN

▶

ICE (Interactive Connectivity Establishment) — фреймворк, який WebRTC використовує для пошуку найкращого шляху між пірами. Він збирає та перевіряє ICE-кандидатів — можливі мережеві точки входу.

ТипЩо робитьКоли потрібен
HostПряма LAN IP/портТа сама мережа
Server Reflexive (srflx)Публічна IP через STUNЗа NAT (більшість випадків)
Relay (relay)Трафік через TURN-серверСиметричний NAT / файрвол

STUN (Session Traversal Utilities for NAT) — дешевий, stateless-сервер. Повідомляє вашу публічну IP:порт, щоб ви могли поділитися нею як кандидатом.

TURN (Traversal Using Relays around NAT) — резервний relay-сервер, що пересилає весь медіатрафік. Дорогий, але завжди працює.

const iceConfig = {
  iceServers: [
    { urls: 'stun:stun.l.google.com:19302' },
    {
      urls: 'turn:turn.example.com:3478',
      username: 'user',
      credential: 'secret',
    },
  ],
  iceTransportPolicy: 'all', // 'relay' — лише через TURN
};
~80% дзвінків з'єднуються через STUN (srflx). Завжди розгортайте TURN як резерв — симетричний NAT і корпоративні файрволи потребують його.
SDP — Session Description Protocol Протокол

Структура SDP та ключові поля

▶

SDP — текстовий формат (не протокол), що описує медіасесії: кодеки, пропускну здатність, напрямок, облікові дані ICE, відбиток DTLS. Передається у WebRTC offer/answer.

# Секція сесії
v=0                              # версія
o=- 123456 2 IN IP4 127.0.0.1   # origin
s=-                              # назва сесії
t=0 0                            # час (0 0 = постійно)

# Медіасекція (аудіо)
m=audio 9 UDP/TLS/RTP/SAVPF 111  # порт 9 = bundled
a=rtpmap:111 opus/48000/2        # кодек Opus, 48kHz стерео
a=fmtp:111 minptime=10;useinbandfec=1
a=sendrecv                        # напрямок
a=ice-ufrag:Xd3r                  # облікові дані ICE
a=ice-pwd:longpassword
a=fingerprint:sha-256 AA:BB:...  # відбиток DTLS-сертифіката
a=setup:actpass
Атрибут напрямкуЗначення
sendrecvНадсилає та отримує (за замовчуванням)
sendonlyЛише надсилає (джерело демонстрації екрана)
recvonlyЛише отримує (глядач)
inactiveНічого (на утриманні)
BUNDLE об'єднує кілька медіапотоків через один транспорт — одне ICE/DTLS-з'єднання, кілька m-секцій. Сучасний WebRTC завжди його використовує.

// Надаємо перевагу Opus, видаляємо інші аудіокодеки
function preferOpus(sdp) {
  const lines = sdp.split('\r\n');
  const opusLine = lines.find(l => l.includes('opus/48000'));
  const payload = opusLine?.match(/a=rtpmap:(\d+)/)?.[1];
  if (!payload) return sdp;
  return sdp.replace(/m=audio (\d+) [^\r]+/, `m=audio $1 UDP/TLS/RTP/SAVPF ${payload}`);
}

// Встановлюємо максимальний бітрейт через RTCRtpSender
const sender = pc.getSenders().find(s => s.track?.kind === 'video');
const params = sender.getParameters();
params.encodings[0].maxBitrate = 1_000_000; // 1 Mbps
await sender.setParameters(params);
SIP та VoIP Протокол

Огляд SIP та потік виклику

▶

SIP (Session Initiation Protocol, RFC 3261) — текстовий протокол сигналізації для VoIP. Відповідає за встановлення, зміну та завершення сесій. Сам медіатрафік передається через RTP — SIP лише узгоджує його.

Ключові сутності SIP:

СутністьРоль
UA (User Agent)Клієнт/телефон, що ініціює або приймає дзвінки
Proxy ServerМаршрутизує SIP-повідомлення між UA
RegistrarВідображає SIP URI → поточний IP:порт (через REGISTER)
B2BUABack-to-back UA — завершує і повторно ініціює дзвінки (Asterisk, FreeSWITCH)
# Базовий SIP INVITE (вихідний дзвінок)
INVITE sip:bob@example.com SIP/2.0
Via: SIP/2.0/UDP alice-pc:5060;branch=z9hG4bK776asdhds
From: "Alice" <sip:alice@example.com>;tag=1928301774
To: <sip:bob@example.com>
Call-ID: a84b4c76e66710@pc33.atlanta.com
CSeq: 314159 INVITE
Content-Type: application/sdp
Content-Length: [sdp length]

[SDP тіло — узгодження кодеків]

Базовий потік виклику:

  1. INVITE → 100 Trying → 180 Ringing → 200 OK → ACK (триетапне рукостискання)
  2. Передача RTP-медіа
  3. BYE → 200 OK (завершення)
SIP використовує SDP для узгодження медіа — тіло INVITE містить SDP offer, 200 OK — SDP answer. Та сама модель offer/answer, що і в WebRTC.

RTP та RTCP

▶

RTP (Real-time Transport Protocol) переносить медіа (аудіо/відео) через UDP. RTCP — супутній протокол управління для статистики QoS.

RTPRTCP
ПризначенняТранспорт медіаЗворотній зв'язок QoS та синхронізація
ПортПарний (напр. 5004)Непарний = порт RTP + 1
ТранспортUDP (зазвичай)UDP
Ключові поляSeq#, мітка часу, SSRC, тип навантаженняSR/RR звіти, SDES, BYE, APP

SRTP (Secure RTP) — обов'язковий у WebRTC, шифрує медіанавантаження та автентифікує заголовки. Обмін ключами через DTLS-SRTP.

WebRTC вимагає SRTP — звичайний RTP не допускається. DTLS-рукостискання отримує головний ключ SRTP. Відбиток у SDP автентифікує сертифікат.
Janus WebRTC Server Сервер

Архітектура Janus та система плагінів

▶

Janus — відкритий WebRTC серверний шлюз (від Meetecho). Виступає як selective forwarding unit (SFU) та B2BUA — завершує WebRTC з одного боку та може з'єднуватись з SIP, RTP, записами тощо через плагіни.

ПлагінСценарій використання
VideoRoomБагатосторонні відеоконференції (SFU)
VideoCallПростий відеодзвінок 1:1
SIPМіст WebRTC ↔ SIP/PSTN
StreamingТрансляція RTP/RTSP-потоків до WebRTC-глядачів
AudioBridgeАудіо MCU — змішує кілька аудіопотоків
RecordPlayЗапис і відтворення WebRTC-сесій
TextRoomТекстові повідомлення через DataChannels

Транспортні рівні: HTTP REST, WebSocket, RabbitMQ, MQTT — всі використовують однаковий JSON-based Janus API.

Потік Janus API

▶

Кожна взаємодія з Janus: створити сесію → підключити хендл плагіна → надіслати повідомлення плагіна + JSEP.

// 1. Створюємо Janus-сесію
const janus = new Janus({
  server: 'wss://janus.example.com/ws',
  success: () => {
    // 2. Підключаємось до плагіна VideoRoom
    janus.attach({
      plugin: 'janus.plugin.videoroom',
      success: (handle) => { pluginHandle = handle; },
      onmessage: (msg, jsep) => {
        if (jsep) {
          pluginHandle.createAnswer({
            jsep,
            tracks: [{ type: 'audio', capture: true }, { type: 'video', capture: true }],
            success: (ourJsep) => pluginHandle.send({ message: { request: 'start' }, jsep: ourJsep }),
          });
        }
      },
      onremotetrack: (track, mid, active) => {
        if (active) {
          const stream = new MediaStream([track]);
          videoEl.srcObject = stream;
        }
      },
    });
  },
});

// 3. Заходимо до кімнати
pluginHandle.send({ message: { request: 'join', room: 1234, ptype: 'publisher', display: 'Alice' } });
Janus обробляє ICE, DTLS та SRTP всередині. Ваш застосунок працює лише з JSON-повідомленнями та JSEP (JSON-обгортки SDP) — Janus перекладає до/від нативного WebRTC.

Основи мереж для VoIP Мережі

Типи NAT та їх обхід

▶

NAT (Network Address Translation) відображає приватні IP на публічну IP. Різна поведінка NAT впливає на можливість прямого з'єднання між пірами.

Тип NATПоведінкаP2P можливий?
Full ConeБудь-який зовнішній хост може надіслати на відображений портТак (легко)
Restricted ConeЛише хости, яким ви надсилали, можуть відповістиТак (STUN)
Port-Restricted ConeIP + порт мають збігатисяТак (STUN)
SymmetricРізне відображення для кожного призначенняПотрібен TURN

ICE trickle — кандидати надсилаються віддаленому піру в міру їх збору, не чекаючи всіх. Прискорює встановлення з'єднання.

Симетричний NAT (поширений у корпоративних мережах) блокує обхід лише через STUN. Завжди надавайте TURN-сервери для production.

Ключові механізми управління перевантаженням WebRTC:

  • REMB — Receiver Estimated Max Bitrate (старіший, RTCP зворотній зв'язок)
  • TWCC — Transport-Wide Congestion Control (на стороні відправника, точніший)
  • Simulcast — видавець надсилає кілька роздільних здатностей; SFU обирає для кожного підписника
  • SVC — Scalable Video Coding — пошарове кодування (VP9/AV1 підтримують)
