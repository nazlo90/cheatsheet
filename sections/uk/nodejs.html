Node.js Event Loop & I/O Основне

libuv, фази та event loop Node

▶

Node.js запускає JS на V8 і делегує асинхронний I/O до libuv, який використовує механізми ОС (epoll, kqueue, IOCP). Event loop має шість фаз, що виконуються по порядку за кожен тік:

ФазаЩо тут виконується
timersКоллбеки setTimeout / setInterval, чий час вийшов
pending callbacksПомилки I/O, відкладені з попереднього тіку
idle / prepareВнутрішнє використання libuv
pollНові I/O-події; блокується тут якщо черга порожня (до дедлайну таймера)
checkКоллбеки setImmediate
close callbackssocket.on('close', …)

Між кожним переходом фаз Node спочатку спустошує чергу process.nextTick, потім мікрозавдання Promise (так само як мікрозавдання браузера).

setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));
process.nextTick(() => console.log('nextTick'));
Promise.resolve().then(() => console.log('promise'));
// nextTick → promise → timeout → immediate
// (порядок timeout vs immediate є недетермінованим поза I/O)
process.nextTick спрацьовує до Promise і може заблокувати event loop при рекурсивному виклику. Надавайте перевагу queueMicrotask() для мікрозавдань у користувацькому коді.

Патерни неблокуючого I/O

▶

Node досягає конкурентності без потоків, делегуючи I/O ОС і використовуючи коллбеки/проміси після завершення роботи.

import fs from 'node:fs/promises';

// Паралельне читання — не чекайте послідовно!
const [a, b] = await Promise.all([
  fs.readFile('a.txt', 'utf8'),
  fs.readFile('b.txt', 'utf8'),
]);

// Потокове читання великих файлів замість завантаження в пам'ять
import { createReadStream } from 'node:fs';
createReadStream('huge.csv').pipe(processStream).pipe(res);
CPU-інтенсивна робота (обробка зображень, шифрування, стиснення) блокує event loop — виносьте її у потік Worker або дочірній процес.

▶

Поле "exports" замінює "main" та "module". Глибокі імпорти, не зазначені тут, блокуються — добре для інкапсуляції.

Потоки (Streams) & Буфери Продуктивність

Типи потоків та зворотний тиск (back-pressure)

▶

Потоки обробляють дані частинами, уникаючи завантаження цілих файлів у пам'ять. Node має чотири типи потоків:

ТипПрикладКлючові методи
Readablefs.createReadStream(), reqpipe(), read(), подія 'data'
Writablefs.createWriteStream(), reswrite(), end()
DuplexTCP сокетиВодночас Readable і Writable
Transformzlib.createGzip()Модифікує дані під час передачі
import { pipeline } from 'node:stream/promises';
import { createReadStream, createWriteStream } from 'node:fs';
import { createGzip } from 'node:zlib';

// pipeline автоматично обробляє back-pressure та очищення
await pipeline(
  createReadStream('input.log'),
  createGzip(),
  createWriteStream('input.log.gz')
);
Back-pressure: коли writable не встигає, write() повертає false. Призупиніть readable до спрацювання 'drain'. pipeline() обробляє це автоматично — використовуйте його замість raw .pipe().

Буфери та кодування

▶
// Buffer = блок сирої пам'яті фіксованого розміру (uint8)
const buf = Buffer.from('hello', 'utf8');
buf.toString('hex');   // '68656c6c6f'
buf.toString('base64'); // 'aGVsbG8='

Buffer.alloc(10);       // безпечне виділення з нулями
Buffer.allocUnsafe(10); // швидше, але може містити старі дані

// Безпечне перетворення частин потоку в рядок
const chunks = [];
stream.on('data', c => chunks.push(c));
stream.on('end', () => { const result = Buffer.concat(chunks).toString(); });
Завжди вказуйте кодування при виклику toString(). За замовчуванням UTF-8. Використовуйте TextDecoder для інкрементного декодування багатобайтових символів, розбитих між частинами.
HTTP & Express Практично

Чистий модуль http vs Express

▶
// Чистий Node HTTP
import http from 'node:http';
http.createServer((req, res) => {
  res.writeHead(200, { 'Content-Type': 'application/json' });
  res.end(JSON.stringify({ ok: true }));
}).listen(3000);

// Еквівалент на Express
import express from 'express';
const app = express();
app.use(express.json());

app.get('/users/:id', async (req, res, next) => {
  try {
    const user = await db.findById(req.params.id);
    if (!user) return res.status(404).json({ error: 'Not found' });
    res.json(user);
  } catch(e) { next(e); }
});

// Middleware для обробки помилок (4 аргументи)
app.use((err, req, res, next) => {
  res.status(500).json({ error: err.message });
});
Завжди викликайте next(err) в асинхронних обробниках маршрутів — необроблені відхилення промісів не потраплять до middleware помилок. Express 5 обробляє це автоматично.

Ланцюг middleware та поширені патерни

▶
// Middleware = (req, res, next) => void

// Middleware автентифікації
function auth(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'Unauthorized' });
  req.user = verifyToken(token);
  next();
}

// Групування роутерів
const router = express.Router();
router.use(auth);
router.get('/', listUsers);
router.post('/', createUser);
app.use('/api/users', router);

// Обмеження запитів (express-rate-limit)
import rateLimit from 'express-rate-limit';
app.use(rateLimit({ windowMs: 15*60*1000, max: 100 }));
Порядок middleware має значення. Розміщуйте express.json(), CORS та rate-limit перед маршрутами. Обробник помилок — останнім.

Cluster vs Worker Threads vs child_process

▶
APIСценарій використанняПам'ятьКомунікація

worker_threadsCPU-інтенсивна JS-робота (шифрування, парсинг)Спільний ArrayBufferpostMessage / SharedArrayBuffer
child_processЗапуск зовнішніх програм / shell-командОкремі процесиstdin/stdout/IPC
// Cluster — первинний форкає по одному воркеру на ядро CPU
import cluster from 'node:cluster';
import os from 'node:os';

if (cluster.isPrimary) {
  os.cpus().forEach(() => cluster.fork());
  cluster.on('exit', (w) => { console.log(`Worker ${w.process.pid} died`); cluster.fork(); });
} else {
  // кожен воркер запускає HTTP-сервер незалежно
  startServer();
}

Worker Threads для CPU-інтенсивних задач

▶
// worker.js
import { workerData, parentPort } from 'node:worker_threads';
const result = heavyCompute(workerData);
parentPort.postMessage(result);

// main.js
import { Worker } from 'node:worker_threads';

function runWorker(data) {
  return new Promise((resolve, reject) => {
    const w = new Worker('./worker.js', { workerData: data });
    w.on('message', resolve);
    w.on('error', reject);
  });
}

// Transferable об'єкти — передача буфера без копіювання
const buf = new SharedArrayBuffer(1024);
w.postMessage({ buf }, [buf]); // передаємо власність, не копіюємо
Використовуйте бібліотеку пулу воркерів (напр. piscina) замість ручного управління. Створення воркера на кожен запит — дороге задоволення.
npm, Пакети & Безпека Ops

Основи npm та lock-файли

▶
# Управління залежностями
npm install                  # встановити з package-lock.json
npm ci                       # чиста установка — швидша, сувора, безпечна для CI
npm install pkg --save-exact # зафіксувати точну версію
npm outdated                 # показати застарілі залежності
npm update                   # оновити в межах діапазону semver

# Аудит
npm audit                    # звіт про відомі вразливості
npm audit fix                # автовиправлення сумісних оновлень
npm audit fix --force        # може внести критичні зміни

# Монорепо workspace
npm install --workspace=packages/app
ДіапазонЗначенняПриклад
^1.2.3Сумісний (minor+patch)≥1.2.3 <2.0.0
~1.2.3Лише patch≥1.2.3 <1.3.0
1.2.3Точна версія1.2.3 лише
Завжди комітьте package-lock.json. Використовуйте npm ci в CI-пайплайнах — він завершується з помилкою, якщо lock-файл не синхронізований з package.json.

Найкращі практики безпеки

▶
  • Ланцюжок постачання: фіксуйте залежності, використовуйте npm audit у CI, розгляньте socket.dev для поведінкового аналізу
  • Секрети: ніколи не комітьте .env; використовуйте змінні середовища, що вводяться під час виконання
  • Prototype pollution: перевіряйте/санітизуйте ненадійний JSON перед злиттям в об'єкти; використовуйте Object.create(null) для простих словників
  • Path traversal: завжди використовуйте path.resolve() та перевіряйте, що розрахований шлях знаходиться в межах очікуваного кореня
  • eval / child_process: ніколи не передавайте несанітизований ввід користувача до eval, exec або Function()
  • HTTPS: примусово використовуйте TLS, встановлюйте заголовки безпеки (helmet), впроваджуйте дозволений список CORS
// Захист від path traversal
const root = path.resolve('./public');
const target = path.resolve(root, userInput);
if (!target.startsWith(root + path.sep)) throw new Error('Forbidden');

// helmet — встановлює заголовки безпеки в одному рядку
import helmet from 'helmet';
app.use(helmet());
Запускайте Node від імені непривілейованого користувача в контейнерах. Встановіть --max-old-space-size, щоб запобігти OOM від використання всієї пам'яті хоста.
