Як працюють браузери — Навігація Основне

DNS → TCP → TLS → Перший байт

▶

Коли користувач переходить за URL, браузер проходить кілька мережевих кроків до отримання першого HTML. Розуміння цього конвеєра є ключем до діагностики та оптимізації продуктивності завантаження.

КрокЩо відбуваєтьсяВартість
DNS-запитПеретворює ім'я хоста на IP-адресу (наприклад example.com → 93.184.216.34)Кешується після першого запиту; на мобільних пристроях додає значну затримку на кожне унікальне ім'я хоста
Рукостискання TCPSYN → SYN-ACK → ACK встановлює з'єднання та узгоджує параметри3 повідомлення = 3 кола передачі
Переговори TLSВизначає шифр, перевіряє сертифікат, встановлює зашифрований каналЩе 5 кіл передачі — 8 загалом до будь-яких даних
HTTP GETБраузер надсилає запит; сервер відповідає першим пакетомПерший фрагмент ≈ 14 КБ (обмеження TCP slow start)
Штраф на мобільних: кожне унікальне ім'я хоста (шрифти, скрипти, CDNи, реклама) вимагає власного DNS-запиту — мінімізація кількості різних доменів зменшує затримку.

TCP Slow Start і TTFB

▶

Час до першого байту (TTFB) вимірює проміжок між запитом користувача та отриманням першого пакету даних — ключовий ранній показник продуктивності сервера та мережі.

TCP Slow Start / контроль перевантаження визначає швидкість передачі даних:

  • Вікно перевантаження (CWND) починається з 1–10 MSS (≈ 1500 байт на сегмент для Ethernet)
  • Кожен отриманий ACK → CWND подвоюється (експоненційне зростання)
  • Втрата пакету / відсутність ACK → CWND зменшується вдвічі (експоненційне зменшення)
  • Результат: перші 14 КБ надходять за одне коло передачі; наступні дані ростуть поступово
Помістіть найкритичніший контент «вище згину» в перші 14 КБ HTML-відповіді, щоб використати перше TCP-вікно та забезпечити ранній First Contentful Paint.
Як працюють браузери — Розбір

Побудова DOM і CSSOM

▶

При надходженні HTML браузерний парсер токенізує його та будує дерево DOM. Паралельно CSS розбирається в окреме дерево CSSOM (CSS Object Model). Обидва дерева необхідні до початку рендерингу.

DOMCSSOM
ВхідБайти HTMLБайти CSS
ВихідДерево вузлів, що представляє вміст і ієрархіюДерево правил з обчисленими стилями
Блокує?Блокується синхронним <script>Блокує рендеринг (не розбір)
ШвидкістьІнкрементнаДуже швидко — рідко є вузьким місцем
/* CSSOM застосовує каскад: від загального до конкретного */
body { font-size: 16px; }
h1   { font-size: 2em; }   /* успадковує body, перевизначає розмір */
h1.hero { color: red; }   /* більш специфічне — застосовується останнім */

Розбір JavaScript і блокування рендерингу

▶

Тег <script> без async або defer блокує і розбір HTML, і рендеринг. Браузер повинен завантажити, розібрати, скомпілювати та виконати скрипт перед продовженням.

АтрибутЗавантаженняВиконанняБлокує парсер?
(відсутній)СинхроннеОдразуТак
asyncПаралельнеЩойно готовийТільки при виконанні
deferПаралельнеПісля розбору DOMНі
<!-- Блокує розбір: -->
<script src="heavy.js"></script>

<!-- Не блокує: -->
<script src="analytics.js" async></script>
<script src="app.js" defer></script>
CSS також опосередковано блокує виконання скриптів — браузер не запускатиме <script>, поки CSSOM не готовий, оскільки скрипти можуть запитувати обчислені стилі.

Сканер попереднього завантаження

▶

Поки основний потік заблоковано (наприклад, виконує скрипт), вторинний сканер попереднього завантаження спекулятивно розбирає сирий HTML і завчасно ініціює мережеві запити для пріоритетних ресурсів — CSS, скриптів, шрифтів, зображень — ще до того, як до них дійде основний парсер.

<!-- Сканер попереднього завантаження завантажить styles.css і обидва скрипти,
     навіть поки основний парсер заблокований попереднім скриптом -->
<link rel="stylesheet" href="styles.css">
<script src="app.js" async></script>
<img src="hero.jpg" alt="Hero">
<script src="widget.js" async></script>

Сканер попереднього завантаження — ось чому переміщення скриптів у кінець <body> або використання async/defer все одно дозволяє паралельне виявлення ресурсів.

Як працюють браузери — Критичний шлях рендерингу

6-кроковий конвеєр рендерингу

▶

Критичний шлях рендерингу — це послідовність кроків, які браузер виконує для перетворення HTML, CSS та JS у пікселі на екрані. Його оптимізація безпосередньо покращує First Contentful Paint та інтерактивність.

#КрокОпис
1СтиліОб'єднання DOM + CSSOM у дерево рендерингу — виключає вузли display:none і <head>; включає visibility:hidden (все ще займає місце)
2РозкладкаОбчислення точного положення та розміру кожного вузла в дереві рендерингу (перший запуск = layout; наступні = reflow)
3МалюванняПеретворення блоків на піксельні інструкції: текст, кольори, рамки, тіні. Ціль 60 FPS → ≤ 16.67 мс на кадр
4КомпозитингКоли елементи перекриваються між шарами, об'єднує їх у правильному порядку — часто з прискоренням GPU
/* Запуск кожного етапу має різну вартість */
el.style.width = '200px';   // Layout + Paint + Composite (дорого)
el.style.color = 'red';     // Paint + Composite
el.style.transform = 'translateX(10px)'; // Тільки Composite (найдешевше)

Шари і композитинг

▶

Браузер автоматично підвищує певні елементи до власного шару композитора, що малюється на GPU. Це дозволяє уникнути перемальовування всієї сторінки та звільняє основний потік.

Елементи, що отримують власний шар:

  • <video>, <canvas>
  • opacity, 3D transform
  • will-change: transform (явна підказка)
  • Елементи з предком position: fixed
/* Підвищення до шару GPU для плавної анімації */
.animated {
  will-change: transform;  /* підказка браузеру */
  transform: translateZ(0); /* також примусово створює шар (хак, краще will-change) */
}
/* Прибрати підказку після завершення анімації, щоб звільнити пам'ять */
el.style.willChange = 'auto';
Кожен шар споживає пам'ять GPU. Підвищення всього є контрпродуктивним — використовуйте шари лише для елементів, що фактично анімуються або потребують ізоляції.

Reflow і repaint — що їх викликає

▶

Reflow (Layout) — найдорожча операція: вона змушує браузер перераховувати позиції потенційно для всього документа.

ТригерЗапускає етап
Зміна width, height, margin, padding, font-sizeLayout → Paint → Composite
Зміна color, background, box-shadowPaint → Composite
Зміна transform, opacity (на композитному шарі)Тільки Composite
Читання offsetWidth, getBoundingClientRect()Примусовий синхронний layout (layout thrashing)
// Layout thrashing — уникайте
elements.forEach(el => {
  const width = el.offsetWidth; // примусовий layout
  el.style.width = width * 2 + 'px'; // інвалідує layout → цикл повторюється
});

// Правильно — групуйте читання, потім записи
const widths = elements.map(el => el.offsetWidth); // один layout
elements.forEach((el, i) => el.style.width = widths[i] * 2 + 'px');

// Також: визначайте розміри зображень, щоб запобігти reflow при завантаженні
<img src="photo.jpg" width="800" height="600" alt="...">
Як працюють браузери — Інтерактивність і метрики продуктивності

Time to Interactive і основний потік

▶

Time to Interactive (TTI) вимірює, коли сторінка може надійно реагувати на введення користувача протягом 50 мс після First Contentful Paint. Сторінка може виглядати готовою, але залишатися непідйомною, якщо основний потік зайнятий.

Блокувальники основного потоку після DOMContentLoaded:

  • Розбір, компіляція та виконання JavaScript (включаючи відкладені скрипти)
  • Довгі завдання (> 50 мс) перешкоджають обробці введення
  • Важкі синхронні ланцюжки layout/reflow
// Розбивайте довгі завдання за допомогою scheduler або setTimeout
async function processLargeList(items) {
  for (let i = 0; i < items.length; i++) {
    processItem(items[i]);
    if (i % 100 === 0) await new Promise(r => setTimeout(r, 0)); // передаємо керування
  }
}
Пакет JavaScript розміром 2 МБ на повільному з'єднанні означає, що користувач бачить відрендерену сторінку, але не може взаємодіяти — прокрутка зависає, натискання ігноруються. Відкладайте та розбивайте на частини некритичний JS.

Конвеєр рендерингу браузера — повне резюме

▶
Навігація
  └─ DNS-запит  →  TCP-рукостискання  →  TLS-переговори
       └─ HTTP GET  →  TTFB  →  перший фрагмент 14 КБ

Розбір
  ├─ HTML  →  дерево DOM
  ├─ CSS   →  дерево CSSOM         (блокує рендеринг)
  ├─ JS    →  розбір / компіляція  (блокує розбір якщо синхронний)
  └─ AOM (дерево доступності)   (семантичне дзеркало DOM)

Рендеринг (Критичний шлях рендерингу)
  ├─ Стилі    →  Дерево рендерингу  (DOM + CSSOM, тільки видимі вузли)
  ├─ Layout   →  геометрія блоків  (reflow при змінах)
  ├─ Paint    →  піксельні інструкції на шар
  └─ Composite  →  злиття GPU, правильний z-порядок

Інтерактивність
  └─ TTI: основний потік вільний, реагує на введення < 50 мс

Контрольний список оптимізації:

  • Об'єднуйте домени для зменшення DNS-запитів
  • Вміщайте критичний контент у перші 14 КБ HTML
  • Використовуйте async/defer для некритичних скриптів
  • Завжди встановлюйте width/height на зображення для запобігання зміщенню layout
  • Надавайте перевагу анімаціям transform/opacity (тільки composite)
  • Групуйте читання DOM перед записами, щоб уникнути layout thrashing
  • Використовуйте will-change помірно для GPU-прискорених шарів
  • Розбивайте на частини та відкладайте великі JS-бандли для покращення TTI
