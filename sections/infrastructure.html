CI/CD DevOps

CI/CD concepts

▶

CI (Continuous Integration) — automatically build and test every push/PR. Catches issues early. CD (Continuous Delivery/Deployment) — automatically deploy to environments when CI passes.

StageWhat happens
TriggerPush to branch / PR opened / tag created
Installnpm ci — clean, reproducible install
LintESLint, Stylelint — code quality
TestUnit tests (Jasmine/Vitest), coverage report
Buildng build --prod — production bundle
E2EPlaywright against staging
DeployPush to staging / prod (Kubernetes, S3, etc.)
NotifySlack/email on success or failure

GitHub Actions — Angular example

▶
# .github/workflows/ci.yml
name: CI/CD Pipeline

on:
  push:
    branches: [main, develop]
  pull_request:
    branches: [main]

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-node@v4
        with:
          node-version: '20'
          cache: 'npm'

      - run: npm ci
      - run: npm run lint
      - run: npm run test -- --no-watch --browsers=ChromeHeadless --coverage
      - run: npm run build -- --configuration=production

      - name: Upload coverage
        uses: codecov/codecov-action@v3

  deploy-staging:
    needs: build-and-test
    if: github.ref == 'refs/heads/develop'
    runs-on: ubuntu-latest
    steps:
      - name: Deploy to staging
        run: |
          aws s3 sync dist/app s3://my-app-staging
          aws cloudfront create-invalidation --distribution-id ${{ secrets.CF_ID }} --paths "/*"

  image: $NODE_IMAGE

  stage: test
  image: $NODE_IMAGE
  script:
    - npm run lint
    - npm test -- --no-watch --browsers=ChromeHeadless
  coverage: '/Statements.*?(\d+(?:\.\d+)?)%/'

build:
  stage: build
  image: $NODE_IMAGE
  script: npm run build -- --configuration=production
  artifacts:
    paths: [dist/]
    expire_in: 1 week

deploy_staging:
  stage: deploy
  script: ./scripts/deploy-staging.sh
  environment: staging
  only: [develop]

CI/CD best practices for frontend

▶
  • npm ci instead of npm install — uses exact lockfile, faster, reproducible
  • Cache node_modules by lockfile hash — skip install if unchanged
  • Run tests in headless Chrome — --browsers=ChromeHeadless
  • Fail fast — lint before tests, tests before build
  • Separate staging and production — deploy staging on merge to develop, prod on tag/release
  • Environment variables via secrets — never commit API keys
  • Nx affected — in monorepo, only test/build changed packages
  • Bundle size budget — configure in angular.json, fail CI if exceeded
// angular.json — bundle budget
"budgets": [
  { "type": "initial", "maximumWarning": "500kb", "maximumError": "1mb" },
  { "type": "anyComponentStyle", "maximumWarning": "2kb" }
]
Monitoring Must Know

Three pillars: Metrics · Logs · Traces

▶

Observability = ability to understand system state from its outputs. Three complementary signals:

PillarWhat it answersExamples
MetricsIs the system healthy? (numbers over time)LCP, error rate, memory usage, API latency
LogsWhat happened? (events with context)JS errors, user actions, API responses
TracesWhere did time go? (request lifecycle)Click → API call → DB → render timeline
Monitoring typeDescriptionTools
RUM (Real User Monitoring)Data from real users' browsers in productionSentry, DataDog RUM, New Relic Browser
SyntheticScripted tests from controlled environments (no real users needed)Playwright, Checkly, Lighthouse CI
Error TrackingCapture, group, alert on JS exceptionsSentry, Bugsnag, Rollbar
APM (App Performance)Backend + frontend correlationDataDog, New Relic, Dynatrace
RUM tells you what real users experience. Synthetic tests run 24/7 before users hit issues. Use both.

Core Web Vitals — the metrics that matter

▶

Google's user-centric performance metrics. Affect SEO ranking and user experience.

MetricMeasuresGoodPoor
LCP — Largest Contentful PaintLoading — when main content is visible< 2.5s> 4s
INP — Interaction to Next PaintResponsiveness — UI reacts to input< 200ms> 500ms
CLS — Cumulative Layout ShiftVisual stability — elements don't jump< 0.1> 0.25
TTFB — Time to First ByteServer response speed< 800ms> 1.8s
FCP — First Contentful PaintFirst pixel rendered< 1.8s> 3s
// Measure Core Web Vitals in code
import { onLCP, onINP, onCLS, onFCP, onTTFB } from 'web-vitals';

onLCP(metric => sendToAnalytics({ name: metric.name, value: metric.value }));
onINP(metric => sendToAnalytics({ name: metric.name, value: metric.value }));
onCLS(metric => sendToAnalytics({ name: metric.name, value: metric.value }));

// Also available via PerformanceObserver (native browser API)
new PerformanceObserver((list) => {
  list.getEntries().forEach(entry => console.log(entry));
}).observe({ type: 'largest-contentful-paint', buffered: true });
INP replaced FID as a Core Web Vital in 2024. FID measured only the first interaction; INP measures all interactions throughout the page lifetime.

performance.mark('checkout-start');
// ... do stuff
performance.mark('checkout-end');

const [measure] = performance.getEntriesByName('checkout-duration');
sendMetric({ name: 'checkout.duration', value: measure.duration });

// Angular-specific: track CD performance with profiler (dev only)
// import { enableProdMode } from '@angular/core';
// Use Angular DevTools "Profiler" tab in Chrome
  • p50 vs p95 vs p99 — always monitor percentiles, not averages. A slow p99 affects 1% of users but may signal a real problem.
  • Alerting fatigue — alert on symptoms (user impact), not causes. "Error rate up 3x" > "CPU at 80%".
  • Runbooks — every alert should link to a runbook: what it means, what to check, how to fix.

Dockerfile for Angular app (multi-stage)

▶
# Stage 1 — build
FROM node:20-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci                        # reproducible install
COPY . .
RUN npm run build -- --configuration=production

# Stage 2 — serve (nginx, ~23MB)
FROM nginx:alpine
COPY --from=builder /app/dist/my-app /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
# nginx.conf — handle Angular routing
server {
  listen 80;
  root /usr/share/nginx/html;
  index index.html;

  location / {
    try_files $uri $uri/ /index.html;   # SPA routing
  }

  location /api {
    proxy_pass http://backend:3000;     # proxy to backend
  }

  gzip on;
  gzip_types text/plain application/javascript text/css;
}

Essential Docker commands

▶
# Build & run
docker build -t my-app:latest .
docker run -p 4200:80 my-app:latest
docker run -d --name my-app -p 80:80 my-app:latest

# docker-compose
docker-compose up -d          # start all services
docker-compose down           # stop and remove containers
docker-compose logs -f frontend # follow logs
docker-compose exec frontend sh # shell into container

# Images
docker images
docker pull nginx:alpine
docker push my-registry/my-app:v1.0.0
docker rmi my-app:latest

# Containers
docker ps                     # running containers
docker ps -a                  # all containers
docker logs -f container_id
docker stop container_id
docker rm container_id
Multi-stage builds are essential — keep the final image small (use nginx:alpine instead of node). Never run as root in production (USER nginx). Use .dockerignore to exclude node_modules, dist, .git.
