Functional Programming & Paradigms Key

Programming paradigms overview

▶

A paradigm is a fundamental style of building programs — it shapes how you think about state, control flow, and composition. Most modern languages are multi-paradigm.

ParadigmCore ideaHow you thinkExamples
ImperativeTell the machine how to do it, step by stepSequences of statements that mutate stateEarly C, assembly, classic loops
ProceduralImperative + reusable procedures/functionsGroup steps into named routinesC, Pascal, early JS scripts
Object-Oriented (OOP)Encapsulate state + behavior into objectsModel the domain as interacting objectsJava, C#, Angular services/components
Functional (FP)Transform data through pure functions, avoid side effectsPipelines of transformations; data is immutableHaskell, Elm, RxJS, Redux, modern JS/TS
DeclarativeDescribe what you want, not howExpress intent; hide control flowSQL, HTML, Angular templates, RxJS operators
ReactivePrograms react to data streams over timeAsync event propagation; everything is a streamRxJS, Signals, Redux Observable
JavaScript / TypeScript is multi-paradigm. You freely mix OOP (classes, DI) with FP (pure pipes, immutable NgRx state) and reactive (RxJS) in the same Angular codebase.

FP core concepts

▶
ConceptDefinitionWhy it matters
Pure functionSame inputs → same output; zero side effectsDeterministic, testable, cacheable
ImmutabilityData is never mutated — transformations produce new valuesPrevents shared-state bugs; enables undo/redo; change detection friendliness
Referential transparencyAn expression can be replaced with its value without changing program behaviorEnables memoization, equational reasoning
First-class functionsFunctions are values — storable in variables, passable as arguments, returnableFoundation of HOFs, composition, callbacks
Higher-order functions (HOF)A function that takes or returns another functionmap, filter, reduce, pipe, RxJS operators
Side effectsAny observable change outside the function scope (DOM write, HTTP call, log)Isolate side effects to the edges; keep core logic pure
// Impure — mutates external state
let total = 0;
function addToTotal(n: number) { total += n; }

// Pure — no side effects, same input → same output
function add(a: number, b: number): number { return a + b; }

// Immutability — produce new value instead of mutating
const original = [1, 2, 3];
const updated  = [...original, 4];  // original unchanged

Function composition, currying & partial application

▶

Composition chains functions so that the output of one becomes the input of the next — building complex behavior from small, reusable pieces.

Currying transforms a multi-argument function into a sequence of single-argument functions. Partial application fixes some arguments up front and returns a function for the rest.

// Composition — f(g(x))
const compose = <T>(...fns: Array<(x: T) => T>) =>
  (x: T) => fns.reduceRight((v, f) => f(v), x);

const pipe = <T>(...fns: Array<(x: T) => T>) =>
  (x: T) => fns.reduce((v, f) => f(v), x);

const double  = (n: number) => n * 2;
const addOne  = (n: number) => n + 1;
const transform = pipe(double, addOne);  // (5) → 10 → 11

// Currying
const multiply = (a: number) => (b: number) => a * b;
const triple   = multiply(3);  // partial application — b is still free
triple(4);  // 12

// RxJS pipe is literal function composition
obs$.pipe(
  filter(x => x > 0),
  map(x => x * 2),
  take(5)
);
RxJS pipe(), Angular pipe in templates, and Array.prototype chain methods (.filter().map().reduce()) are all functional composition in disguise.

Functors, monads & practical JS equivalents

▶

These are abstract FP patterns — you already use them daily in JS without the formal names.

ConceptFormal meaningJS / TS equivalentWhat map/flatMap do
FunctorA container with a map that preserves structureArray, Promise (then), Observablemap transforms the inner value, returns same container type
MonadA container with flatMap (chain) to avoid nested wrappingArray.flatMap, Promise.then returning a Promise, RxJS mergeMap/switchMapflatMap unwraps one level before re-wrapping — prevents Array<Array<T>> or Promise<Promise<T>>

// Array as a Functor
[1, 2, 3].map(x => x * 2);  // [2, 4, 6]  — still an Array

// Array as a Monad — flatMap prevents Array<Array>
[[1, 2], [3, 4]].flatMap(x => x);  // [1, 2, 3, 4]

// Promise as a Monad
fetch('/api/user')
  .then(res => res.json())   // returns Promise — chain flattens it
  .then(user => user.name);  // not Promise<Promise<User>>

// RxJS Observable as a Monad
userId$.pipe(
  switchMap(id => this.http.get(`/users/${id}`))  // flatMap in Observable world
);

FP patterns in TypeScript & Angular

▶
// 1. Pure pipe — no side effects, pure transform
@Pipe({ name: 'formatPrice', pure: true })
export class FormatPricePipe implements PipeTransform {
  transform(value: number, currency = 'USD'): string {
    return new Intl.NumberFormat('en-US', { style: 'currency', currency }).format(value);
  }
}

// 2. NgRx reducer — pure function: (state, action) → new state
const cartReducer = createReducer(initialState,
  on(addItem, (state, { item }) => ({
    ...state,
    items: [...state.items, item]  // immutable update
  }))
);

// 3. Selector as memoized pure function
const selectTotal = createSelector(
  selectItems,
  items => items.reduce((sum, i) => sum + i.price, 0)
);

// 4. Typed pipe() helper for data transformation
const getActiveUserNames = pipe(
  (users: User[]) => users.filter(u => u.active),
  (users: User[]) => users.map(u => u.name)
);
Angular's change detection relies on object identity for OnPush. Always return new references from reducers and selectors — never mutate in place. This is FP immutability in practice.

Recursion & tail-call optimization

▶

Recursion replaces explicit loops in FP — a function calls itself with a smaller sub-problem. Pure FP languages rely heavily on it.

Tail-call optimization (TCO): if the recursive call is the last operation, the engine can reuse the current stack frame instead of pushing a new one — avoiding stack overflow. ES6 mandates TCO in strict mode (V8 dropped it; prefer iteration or trampolines for deep recursion in JS).

// Non-tail-recursive — stack grows O(n)
function factorial(n: number): number {
  return n <= 1 ? 1 : n * factorial(n - 1);
}

// Tail-recursive (accumulator pattern)
function factTCO(n: number, acc = 1): number {
  return n <= 1 ? acc : factTCO(n - 1, n * acc);
}

// Safe deep recursion in JS — trampoline pattern
function trampoline<T>(fn: (...args: any[]) => T | (() => T)) {
  return function(...args: any[]) {
    let result: any = fn(...args);
    while (typeof result === 'function') result = result();
    return result;
  };
}

// 1. Custom Event — emitter MFE
window.dispatchEvent(new CustomEvent('mfe:cart:updated', {
  detail: { itemCount: 3 }, bubbles: true
}));

// 1. Custom Event — listener MFE
window.addEventListener('mfe:cart:updated', (e: CustomEvent) => {
  this.cartCount = e.detail.itemCount;
});

// 2. BroadcastChannel — logout across tabs
const channel = new BroadcastChannel('auth');
channel.postMessage({ type: 'logout' });
channel.onmessage = (e) => { if (e.data.type === 'logout') router.navigate(['/login']); };

// 3. Shared singleton service via Module Federation
// webpack.config — mark as singleton shared dep
shared: { '@myorg/auth-state': { singleton: true, strictVersion: true } }
Rule of thumb: use Custom Events for simple notifications, shared singleton for complex reactive state, URL for navigation-driven state. Avoid direct imports between MFEs — that breaks independent deployability.
