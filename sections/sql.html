Running totals & frame clauses

▶
-- Running total with frame
SELECT
    OrderDate, Amount,
    SUM(Amount) OVER (
        ORDER BY OrderDate
        ROWS BETWEEN UNBOUNDED PRECEDING AND CURRENT ROW
    ) AS RunningTotal,
    -- 3-day moving average
    AVG(Amount) OVER (
        ORDER BY OrderDate
        ROWS BETWEEN 2 PRECEDING AND CURRENT ROW
    ) AS MovingAvg3
FROM Orders;
CTEs & Subqueries Interview

CTE (Common Table Expression)

▶
-- Multiple CTEs chained
WITH ActiveCustomers AS (
    SELECT CustomerId, Name
    FROM   Customers
    WHERE  IsActive = 1
),
CustomerRevenue AS (
    SELECT ac.CustomerId, SUM(o.Total) AS Revenue
    FROM   ActiveCustomers ac
    JOIN   Orders o ON o.CustomerId = ac.CustomerId
    GROUP BY ac.CustomerId
)
SELECT * FROM CustomerRevenue WHERE Revenue > 1000;
CTEs improve readability and can be referenced multiple times in the same query. In SQL Server they are NOT materialized by default — the optimizer may inline them.

Recursive CTE

▶
-- Traverse org hierarchy
WITH OrgTree AS (
    -- Anchor: top-level employees
    SELECT EmployeeId, Name, ManagerId, 0 AS Level
    FROM   Employees
    WHERE  ManagerId IS NULL

    UNION ALL

    -- Recursive: subordinates
    SELECT e.EmployeeId, e.Name, e.ManagerId, ot.Level + 1
    FROM   Employees e
    JOIN   OrgTree ot ON ot.EmployeeId = e.ManagerId
)
SELECT * FROM OrgTree ORDER BY Level, Name;

Stored procedures, variables & flow control

▶
CREATE OR ALTER PROCEDURE dbo.usp_GetTopOrders
    @CustomerId INT,
    @TopN       INT = 10
AS
BEGIN
    SET NOCOUNT ON;

    DECLARE @Revenue DECIMAL(18,2);

    SELECT TOP (@TopN)
        OrderId, Total, OrderDate
    FROM   Orders
    WHERE  CustomerId = @CustomerId
    ORDER BY Total DESC;

    IF @@ROWCOUNT = 0
        RAISERROR('No orders found', 16, 1);
END;

Transactions & error handling

▶
BEGIN TRY
    BEGIN TRANSACTION;

    UPDATE Accounts SET Balance -= 100 WHERE Id = 1;
    UPDATE Accounts SET Balance += 100 WHERE Id = 2;

    COMMIT TRANSACTION;
END TRY
BEGIN CATCH
    ROLLBACK TRANSACTION;
    THROW; -- re-raise original error
END CATCH;
Always check XACT_STATE() before ROLLBACK in nested transactions — a -1 state means the transaction is doomed and cannot commit.

▶

    ON Orders (CustomerId)
    INCLUDE (OrderDate, Total);

    ON Orders (OrderDate)

Statistics & plan cache

▶

UPDATE STATISTICS dbo.Orders;

-- Clear plan cache (dev/test only!)
DBCC FREEPROCCACHE;

-- Find expensive queries in plan cache
SELECT TOP 10
    qs.total_worker_time / qs.execution_count AS AvgCPU,
    SUBSTRING(qt.text, (qs.statement_start_offset/2)+1,
        ((qs.statement_end_offset - qs.statement_start_offset)/2)+1) AS QueryText
FROM   sys.dm_exec_query_stats qs
CROSS APPLY sys.dm_exec_sql_text(qs.sql_handle) qt
ORDER BY AvgCPU DESC;

Triggers (T-SQL)

▶
CREATE OR ALTER TRIGGER trg_Orders_AuditUpdate
ON dbo.Orders
AFTER UPDATE
AS
BEGIN
    SET NOCOUNT ON;
    INSERT INTO OrderAudit (OrderId, OldTotal, NewTotal, ChangedAt)
    SELECT d.OrderId, d.Total, i.Total, GETDATE()
    FROM   deleted d
    JOIN   inserted i ON i.OrderId = d.OrderId
    WHERE  d.Total <> i.Total;
END;
Triggers run in the same transaction as the DML. Heavy logic inside triggers can cause lock contention and slow down writes.
Transaction Isolation Levels (SQL Standard)

Anomalies & Isolation Levels

▶

The transaction isolation level defines which anomalies are possible when transactions run concurrently. The higher the level — the safer, but slower.

AnomalyDescription
Dirty ReadA transaction reads uncommitted changes of another transaction
Non-repeatable ReadRe-reading the same row returns a different value (another transaction modified and committed)
Phantom ReadA repeated query returns new rows (another transaction inserted and committed)
Lost UpdateTwo transactions read, modify and write — one update is lost
Isolation LevelDirty ReadNon-repeatable ReadPhantom Read
READ UNCOMMITTEDPossible ✗Possible ✗Possible ✗
READ COMMITTEDProtected ✓Possible ✗Possible ✗
REPEATABLE READProtected ✓Protected ✓Possible ✗
SERIALIZABLEProtected ✓Protected ✓Protected ✓
Defaults: PostgreSQL — READ COMMITTED; SQL Server — READ COMMITTED; MySQL InnoDB — REPEATABLE READ; Oracle — READ COMMITTED.

Details per level + examples

▶
-- READ UNCOMMITTED — sees uncommitted changes (dirty reads)
SET TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;

-- READ COMMITTED — default in most DBMS
-- Two identical SELECTs in the same transaction may return different results
SET TRANSACTION ISOLATION LEVEL READ COMMITTED;
-- T1: SELECT balance FROM accounts WHERE id=1  → 1000
-- T2: UPDATE accounts SET balance=500 WHERE id=1; COMMIT;
-- T1: SELECT balance FROM accounts WHERE id=1  → 500 (non-repeatable read!)

-- REPEATABLE READ — rows read within the transaction do not change
-- But new INSERTs from another transaction may appear — phantom reads
-- Note: PostgreSQL resolves phantom reads via MVCC even at this level
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- SERIALIZABLE — full protection; transactions appear sequential
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;

-- PostgreSQL snapshot isolation
BEGIN;
SET TRANSACTION ISOLATION LEVEL REPEATABLE READ;
SELECT * FROM orders WHERE user_id = 42;
-- we see the snapshot at the transaction start time
COMMIT;

MVCC & Practical Tips

▶

MVCC (Multi-Version Concurrency Control) — PostgreSQL and MySQL InnoDB store multiple row versions instead of read locks. Readers do not block writers and vice versa.

ScenarioRecommended LevelReason
Reports / analytics (read-only)REPEATABLE READ or SERIALIZABLEConsistent snapshot for the entire query
OLTP (orders, payments)READ COMMITTED + explicit locksBalance between safety and performance
Financial operationsSERIALIZABLEProtection against lost updates and phantom reads
Counters / queuesREAD COMMITTED + SELECT FOR UPDATEOptimistic or pessimistic locking
-- Pessimistic lock — locks the row for update
BEGIN;
SELECT balance FROM accounts WHERE id = 1 FOR UPDATE;
UPDATE accounts SET balance = balance - 100 WHERE id = 1;
COMMIT;

-- Optimistic lock — check version before updating
UPDATE accounts
SET balance = 900, version = version + 1
WHERE id = 1 AND version = 5;
-- if 0 rows updated — conflict, retry the transaction
For most OLTP tasks READ COMMITTED + SELECT FOR UPDATE on critical resources is sufficient. Use SERIALIZABLE sparingly — it can significantly reduce throughput under load.
