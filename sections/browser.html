How Browsers Work — Navigation Core

DNS → TCP → TLS → First Byte

▶

When a user navigates to a URL, the browser goes through several network steps before any HTML is received. Understanding this pipeline is key to diagnosing and optimizing load performance.

StepWhat happensCost
DNS LookupResolves hostname to IP address (e.g. example.com → 93.184.216.34)Cached after first lookup; mobile adds significant latency per unique hostname
TCP HandshakeSYN → SYN-ACK → ACK establishes connection and negotiates parameters3 messages = 3 round trips
TLS NegotiationDetermines cipher, verifies certificate, establishes encrypted channel5 more round trips — 8 total before any data
HTTP GETBrowser sends request; server responds with first packetFirst chunk ≈ 14 KB (TCP slow start limit)
Mobile penalty: each unique hostname (fonts, scripts, CDNs, ads) requires its own DNS lookup — minimising the number of distinct domains reduces latency.

TCP Slow Start & TTFB

▶

Time to First Byte (TTFB) measures the gap between the user's request and the arrival of the first data packet — a key early indicator of server and network performance.

TCP Slow Start / Congestion Control governs how fast data can flow:

  • Congestion Window (CWND) starts at 1–10 MSS (≈ 1500 bytes per segment on Ethernet)
  • Each ACK received → CWND doubles (exponential increase)
  • Packet loss / no ACK → CWND halves (exponential decrease)
  • Result: first 14 KB arrives in one round trip; subsequent data grows gradually
Put the most critical above-the-fold content in the first 14 KB of the HTML response to leverage the first TCP window and enable an early First Contentful Paint.
How Browsers Work — Parsing

DOM & CSSOM construction

▶

As HTML arrives, the browser's HTML parser tokenises it and builds the DOM tree. In parallel, CSS is parsed into a separate CSSOM tree (CSS Object Model). Both trees are required before rendering can begin.

DOMCSSOM
InputHTML bytesCSS bytes
OutputNode tree representing content & hierarchyRule tree with computed styles
Blocking?Blocked by synchronous <script>Blocks rendering (not parsing)
SpeedIncrementalVery fast — rarely a bottleneck
/* CSSOM applies cascade: general → specific */
body { font-size: 16px; }
h1   { font-size: 2em; }   /* inherits body, overrides size */
h1.hero { color: red; }   /* more specific — applied last */

JavaScript parsing & render blocking

▶

A <script> tag without async or defer blocks both HTML parsing and rendering. The browser must fetch, parse, compile, and execute the script before resuming.

AttributeDownloadExecutionBlocks parser?
(none)SynchronousImmediatelyYes
asyncParallelAs soon as readyAt execution only
deferParallelAfter DOM parsedNo
<!-- Blocks parsing: -->
<script src="heavy.js"></script>

<!-- Non-blocking: -->
<script src="analytics.js" async></script>
<script src="app.js" defer></script>
CSS also indirectly blocks script execution — the browser will not run a <script> until the CSSOM is ready, because scripts may query computed styles.

Preload scanner

▶

While the main thread is blocked (e.g. executing a script), a secondary preload scanner speculatively parses the raw HTML and dispatches early network requests for high-priority resources — CSS, scripts, fonts, images — before the main parser reaches them.

<!-- Preload scanner will fetch styles.css and both scripts
     even while the main parser is blocked on an earlier script -->
<link rel="stylesheet" href="styles.css">
<script src="app.js" async></script>
<img src="hero.jpg" alt="Hero">
<script src="widget.js" async></script>

The preload scanner is why moving scripts to the end of <body> or using async/defer still allows parallel resource discovery.

How Browsers Work — Critical Rendering Path

6-step rendering pipeline

▶

The Critical Rendering Path is the sequence of steps the browser performs to convert HTML, CSS, and JS into pixels on screen. Optimising it directly improves First Contentful Paint and interactivity.

#StepDescription
1StyleCombine DOM + CSSOM into a render tree — excludes display:none and <head> nodes; includes visibility:hidden (still occupies space)
2LayoutCalculate exact position and size of every node in the render tree (first run = layout; subsequent = reflow)
3PaintConvert boxes to pixel instructions: text, colours, borders, shadows. Targets 60 FPS → ≤ 16.67 ms per frame
4CompositeWhen elements overlap across layers, composite them in the correct order — often GPU-accelerated
/* Triggering each stage costs differently */
el.style.width = '200px';   // Layout + Paint + Composite (expensive)
el.style.color = 'red';     // Paint + Composite
el.style.transform = 'translateX(10px)'; // Composite only (cheapest)

Layers & compositing

▶

The browser automatically promotes certain elements to their own compositor layer, painted on the GPU. This avoids re-painting the entire page and frees the main thread.

Elements that get their own layer:

  • <video>, <canvas>
  • opacity, 3D transform
  • will-change: transform (explicit hint)
  • Elements with a position: fixed ancestor
/* Promote to GPU layer for smooth animation */
.animated {
  will-change: transform;  /* hint to browser */
  transform: translateZ(0); /* also forces layer (hack, prefer will-change) */
}
/* Remove hint after animation ends to free memory */
el.style.willChange = 'auto';
Each layer consumes GPU memory. Promoting everything is counter-productive — use layers only for elements that actually animate or need isolation.

Reflow & repaint — what triggers them

▶

Reflow (Layout) is the most expensive operation — it forces the browser to recalculate positions for potentially the entire document.

TriggerStage triggered
Change width, height, margin, padding, font-sizeLayout → Paint → Composite
Change color, background, box-shadowPaint → Composite
Change transform, opacity (on a composited layer)Composite only
Read offsetWidth, getBoundingClientRect()Forces synchronous layout (layout thrashing)
// Layout thrashing — avoid
elements.forEach(el => {
  const width = el.offsetWidth; // forces layout
  el.style.width = width * 2 + 'px'; // invalidates layout → cycle repeats
});

// Correct — batch reads, then writes
const widths = elements.map(el => el.offsetWidth); // one layout
elements.forEach((el, i) => el.style.width = widths[i] * 2 + 'px');

// Also: define image dimensions to prevent reflow on load
<img src="photo.jpg" width="800" height="600" alt="...">
How Browsers Work — Interactivity & Performance Metrics

Time to Interactive & main thread

▶

Time to Interactive (TTI) measures when the page can reliably respond to user input within 50 ms after First Contentful Paint. A page can look ready but still be unresponsive if the main thread is busy.

Main thread blockers after DOMContentLoaded:

  • JavaScript parsing, compilation, and execution (including deferred scripts)
  • Long tasks (> 50 ms) prevent input handling
  • Heavy synchronous layout/reflow chains
// Break up long tasks with scheduler or setTimeout
async function processLargeList(items) {
  for (let i = 0; i < items.length; i++) {
    processItem(items[i]);
    if (i % 100 === 0) await new Promise(r => setTimeout(r, 0)); // yield
  }
}
A 2 MB JavaScript bundle on a slow connection means the user sees a rendered page but cannot interact — scrolling janks, taps are ignored. Defer and code-split non-critical JS.

Browser rendering pipeline — full summary

▶
Navigation
  └─ DNS lookup  →  TCP handshake  →  TLS negotiation
       └─ HTTP GET  →  TTFB  →  14 KB first chunk

Parsing
  ├─ HTML  →  DOM tree
  ├─ CSS   →  CSSOM tree         (blocks rendering)
  ├─ JS    →  parse / compile    (blocks parsing if sync)
  └─ AOM (Accessibility tree)   (mirrors DOM semantically)

Rendering (Critical Rendering Path)
  ├─ Style   →  Render tree  (DOM + CSSOM, visible nodes only)
  ├─ Layout  →  box geometry  (reflow on changes)
  ├─ Paint   →  pixel instructions per layer
  └─ Composite  →  GPU merge, correct z-order

Interactivity
  └─ TTI: main thread idle, responds to input < 50 ms

Optimisation checklist:

  • Consolidate domains to reduce DNS lookups
  • Fit critical content in first 14 KB of HTML
  • Use async/defer on non-critical scripts
  • Always set width/height on images to prevent layout shift
  • Prefer transform/opacity animations (composite-only)
  • Batch DOM reads before DOM writes to avoid layout thrashing
  • Use will-change sparingly for GPU-accelerated layers
  • Code-split and defer large JS bundles to improve TTI
