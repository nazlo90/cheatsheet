WebRTC Architecture Core

WebRTC Overview & Connection Flow

▶

WebRTC (Web Real-Time Communication) enables peer-to-peer audio, video, and data directly in the browser without plugins. It uses three main APIs: RTCPeerConnection, RTCDataChannel, and getUserMedia.

Connection flow (Offer/Answer):

  1. Caller creates Offer (SDP) via createOffer()
  2. Caller sets it as local description, sends SDP to callee via signaling
  3. Callee sets it as remote description, creates Answer
  4. Callee sets answer as local description, sends back via signaling
  5. Both sides exchange ICE candidates through signaling
  6. ICE connectivity checks complete — P2P media flows
// Caller side
const pc = new RTCPeerConnection(iceConfig);

// Add local tracks before creating offer
const stream = await navigator.mediaDevices.getUserMedia({ audio: true, video: true });
stream.getTracks().forEach(track => pc.addTrack(track, stream));

const offer = await pc.createOffer();
await pc.setLocalDescription(offer);
signalingChannel.send({ type: 'offer', sdp: offer.sdp });

// On ICE candidate gathered
pc.onicecandidate = ({ candidate }) => {
  if (candidate) signalingChannel.send({ type: 'ice', candidate });
};

// Receive remote tracks
pc.ontrack = ({ streams }) => { videoEl.srcObject = streams[0]; };
// Callee side
const pc = new RTCPeerConnection(iceConfig);
await pc.setRemoteDescription(new RTCSessionDescription(offer));
const answer = await pc.createAnswer();
await pc.setLocalDescription(answer);
signalingChannel.send({ type: 'answer', sdp: answer.sdp });

// Add received ICE candidates
await pc.addIceCandidate(new RTCIceCandidate(candidate));

ICE, STUN & TURN

▶

ICE (Interactive Connectivity Establishment) is the framework WebRTC uses to find the best path between peers. It gathers and tests ICE candidates — possible network endpoints.

TypeWhat it doesWhen needed
HostDirect LAN IP/portSame network
Server Reflexive (srflx)Public IP via STUNBehind NAT (most cases)
Relay (relay)Traffic via TURN serverSymmetric NAT / firewall

STUN (Session Traversal Utilities for NAT) — cheap, stateless server. Tells you your public IP:port so you can share it as a candidate.

TURN (Traversal Using Relays around NAT) — fallback relay server that forwards all media. Expensive but always works.

const iceConfig = {
  iceServers: [
    { urls: 'stun:stun.l.google.com:19302' },
    {
      urls: 'turn:turn.example.com:3478',
      username: 'user',
      credential: 'secret',
    },
  ],
  iceTransportPolicy: 'all', // 'relay' forces TURN only
};
~80% of calls connect via STUN (srflx). Always deploy TURN as a fallback — symmetric NAT and corporate firewalls need it.
SDP — Session Description Protocol Protocol

SDP Structure & Key Fields

▶

SDP is a text format (not a protocol) that describes media sessions — codecs, bandwidth, direction, ICE credentials, DTLS fingerprint. Exchanged in the WebRTC offer/answer.

# Session section
v=0                              # version
o=- 123456 2 IN IP4 127.0.0.1   # origin
s=-                              # session name
t=0 0                            # timing (0 0 = permanent)

# Media section (audio)
m=audio 9 UDP/TLS/RTP/SAVPF 111  # port 9 = bundled
a=rtpmap:111 opus/48000/2        # Opus codec, 48kHz stereo
a=fmtp:111 minptime=10;useinbandfec=1
a=sendrecv                        # direction
a=ice-ufrag:Xd3r                  # ICE credentials
a=ice-pwd:longpassword
a=fingerprint:sha-256 AA:BB:...  # DTLS cert fingerprint
a=setup:actpass
Direction attrMeaning
sendrecvSend and receive (default)
sendonlySend only (screenshare source)
recvonlyReceive only (viewer)
inactiveNeither (on hold)
BUNDLE groups multiple media streams over one transport — one ICE/DTLS connection, multiple m-sections. Modern WebRTC always uses it.

// Prefer Opus, remove other audio codecs
function preferOpus(sdp) {
  const lines = sdp.split('\r\n');
  const opusLine = lines.find(l => l.includes('opus/48000'));
  const payload = opusLine?.match(/a=rtpmap:(\d+)/)?.[1];
  if (!payload) return sdp;
  return sdp.replace(/m=audio (\d+) [^\r]+/, `m=audio $1 UDP/TLS/RTP/SAVPF ${payload}`);
}

// Set max bitrate via RTCRtpSender
const sender = pc.getSenders().find(s => s.track?.kind === 'video');
const params = sender.getParameters();
params.encodings[0].maxBitrate = 1_000_000; // 1 Mbps
await sender.setParameters(params);
SIP & VoIP Protocol

SIP Overview & Call Flow

▶

SIP (Session Initiation Protocol, RFC 3261) is a text-based signaling protocol for VoIP. It handles setup, modification, and teardown of sessions. Media itself runs over RTP — SIP just negotiates it.

Key SIP entities:

EntityRole
UA (User Agent)Client/phone that initiates or receives calls
Proxy ServerRoutes SIP messages between UAs
RegistrarMaps SIP URI → current IP:port (via REGISTER)
B2BUABack-to-back UA — terminates and re-originates calls (Asterisk, FreeSWITCH)
# Basic SIP INVITE (outgoing call)
INVITE sip:bob@example.com SIP/2.0
Via: SIP/2.0/UDP alice-pc:5060;branch=z9hG4bK776asdhds
From: "Alice" <sip:alice@example.com>;tag=1928301774
To: <sip:bob@example.com>
Call-ID: a84b4c76e66710@pc33.atlanta.com
CSeq: 314159 INVITE
Content-Type: application/sdp
Content-Length: [sdp length]

[SDP body — codec negotiation]

Basic call flow:

  1. INVITE → 100 Trying → 180 Ringing → 200 OK → ACK (3-way handshake)
  2. RTP media flows
  3. BYE → 200 OK (teardown)
SIP uses SDP for media negotiation — the INVITE body contains an SDP offer, the 200 OK contains the SDP answer. Same offer/answer model as WebRTC.

RTP & RTCP

▶

RTP (Real-time Transport Protocol) carries media (audio/video) over UDP. RTCP is the companion control protocol for QoS stats.

RTPRTCP
PurposeMedia transportQoS feedback & sync
PortEven (e.g. 5004)Odd = RTP port + 1
TransportUDP (usually)UDP
Key fieldsSeq#, timestamp, SSRC, payload typeSR/RR reports, SDES, BYE, APP

SRTP (Secure RTP) — mandatory in WebRTC, encrypts media payload and authenticates headers. Key exchange via DTLS-SRTP.

WebRTC mandates SRTP — plain RTP is not allowed. The DTLS handshake derives the SRTP master key. Fingerprint in SDP authenticates the cert.
Janus WebRTC Server Server

Janus Architecture & Plugin System

▶

Janus is an open-source WebRTC server gateway (by Meetecho). It acts as a selective forwarding unit (SFU) and B2BUA — it terminates WebRTC on one side and can bridge to SIP, RTP, recordings, etc. via plugins.

PluginUse case
VideoRoomMultiparty video conferences (SFU)
VideoCallSimple 1:1 video call
SIPBridge WebRTC ↔ SIP/PSTN
StreamingBroadcast RTP/RTSP streams to WebRTC viewers
AudioBridgeAudio MCU — mixes multiple audio streams
RecordPlayRecord and playback WebRTC sessions
TextRoomText messaging over DataChannels

Transport layers: HTTP REST, WebSocket, RabbitMQ, MQTT — all carry the same JSON-based Janus API.

Janus API Flow

▶

Every Janus interaction follows: create session → attach plugin handle → send plugin messages + JSEP.

// 1. Create Janus session
const janus = new Janus({
  server: 'wss://janus.example.com/ws',
  success: () => {
    // 2. Attach to VideoRoom plugin
    janus.attach({
      plugin: 'janus.plugin.videoroom',
      success: (handle) => { pluginHandle = handle; },
      onmessage: (msg, jsep) => {
        if (jsep) {
          pluginHandle.createAnswer({
            jsep,
            tracks: [{ type: 'audio', capture: true }, { type: 'video', capture: true }],
            success: (ourJsep) => pluginHandle.send({ message: { request: 'start' }, jsep: ourJsep }),
          });
        }
      },
      onremotetrack: (track, mid, active) => {
        if (active) {
          const stream = new MediaStream([track]);
          videoEl.srcObject = stream;
        }
      },
    });
  },
});

// 3. Join a room
pluginHandle.send({ message: { request: 'join', room: 1234, ptype: 'publisher', display: 'Alice' } });
Janus handles all ICE, DTLS, and SRTP internally. Your app only deals with JSON messages and JSEP (JSON SDP wrappers) — Janus translates to/from native WebRTC.

Networking Fundamentals for VoIP Networking

NAT Types & Traversal

▶

NAT (Network Address Translation) maps private IPs to a public IP. Different NAT behaviors affect whether peers can connect directly.

NAT TypeBehaviourP2P possible?
Full ConeAny external host can send to the mapped portYes (easy)
Restricted ConeOnly hosts you sent to can respondYes (STUN)
Port-Restricted ConeIP + port must matchYes (STUN)
SymmetricDifferent mapping per destinationNeeds TURN

ICE trickle — candidates are sent to the remote peer as they are gathered rather than waiting for all of them. Speeds up connection establishment.

Symmetric NAT (common in corporate networks) blocks STUN-only traversal. Always provision TURN servers for production.

Key WebRTC congestion control mechanisms:

  • REMB — Receiver Estimated Max Bitrate (older, RTCP feedback)
  • TWCC — Transport-Wide Congestion Control (sender-side, more accurate)
  • Simulcast — publisher sends multiple resolutions; SFU picks per subscriber
  • SVC — Scalable Video Coding — layered encoding (VP9/AV1 support it)
