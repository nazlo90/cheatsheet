GraphQL Fundamentals Core

Schema, Types & Operations

▶

GraphQL is a query language and runtime for APIs. Clients request exactly the data they need — no over- or under-fetching.

ConceptDescription
SchemaStrongly typed contract between client and server (SDL)
QueryRead-only data fetching
MutationWrite operation (create / update / delete)
SubscriptionReal-time updates via WebSocket
ResolverFunction that returns data for a field
ScalarLeaf types: String, Int, Float, Boolean, ID
# Schema Definition Language (SDL)
type User {
  id:        ID!
  name:      String!
  email:     String!
  posts:     [Post!]!
  createdAt: String
}

type Query {
  user(id: ID!):  User
  users:            [User!]!
}

type Mutation {
  createUser(input: CreateUserInput!): User!
  deleteUser(id: ID!):               Boolean!
}

input CreateUserInput {
  name:  String!
  email: String!
}
! means non-nullable. [Post!]! means a non-nullable list of non-nullable Post objects — both the list and each element are guaranteed.

Client query syntax

▶
# Query with fields selection
query GetUser($id: ID!) {
  user(id: $id) {
    id
    name
    posts {
      title
      publishedAt
    }
  }
}

# Mutation
mutation CreateUser($input: CreateUserInput!) {
  createUser(input: $input) {
    id
    name
  }
}

# Fragments — reusable field sets
fragment UserFields on User {
  id
  name
  email
}

query { users { ...UserFields } }

# Aliases — query same field twice with different args
query {
  alice: user(id: "1") { name }
  bob:   user(id: "2") { name }
}
Resolvers & Context Runtime

Resolver anatomy (code-first with NestJS)

▶
import { Resolver, Query, Mutation, Args, ID,
         ResolveField, Parent, Context } from '@nestjs/graphql';
import { UsersService } from './users.service';
import { User } from './models/user.model';
import { CreateUserInput } from './dto/create-user.input';

@Resolver(() => User)
export class UsersResolver {
  constructor(private usersService: UsersService) {}

  @Query(() => [User], { name: 'users' })
  findAll() { return this.usersService.findAll(); }

  @Query(() => User, { nullable: true })
  user(@Args('id', { type: () => ID }) id: string) {
    return this.usersService.findOne(id);
  }

  @Mutation(() => User)
  createUser(@Args('input') input: CreateUserInput) {
    return this.usersService.create(input);
  }

  // Field resolver — runs for every User returned
  @ResolveField('posts', () => [Post])
  getPosts(@Parent() user: User) {
    return this.postsService.findByAuthor(user.id);
  }
}
Code-first approach generates the SDL automatically from TypeScript decorators. Schema-first uses hand-written SDL and binds resolvers to it.

ObjectType & InputType models

▶
import { ObjectType, Field, ID, InputType } from '@nestjs/graphql';
import { IsEmail, IsString } from 'class-validator';

@ObjectType()
export class User {
  @Field(() => ID)
  id: string;

  @Field()
  name: string;

  @Field({ nullable: true })
  bio?: string;

  @Field(() => [Post])
  posts: Post[];
}

@InputType()
export class CreateUserInput {
  @Field()
  @IsString()
  name: string;

  @Field()
  @IsEmail()
  email: string;
}
Nest's ValidationPipe works on GraphQL args too — add it globally via app.useGlobalPipes and class-validator decorators on @InputType() classes.
DataLoader & N+1 Problem Performance

N+1 problem & DataLoader solution

▶

When a list query resolves nested fields, naive resolvers fire one DB query per parent — the N+1 problem. DataLoader batches and caches these into a single query per request.

// Without DataLoader — N+1
// 1 query for users, then 1 query per user for posts → N+1
@ResolveField('posts', () => [Post])
getPosts(@Parent() user: User) {
  return this.db.query('SELECT * FROM posts WHERE author_id = $1', [user.id]);
}

// With DataLoader — 1 batched query
import DataLoader from 'dataloader';

export function createPostsLoader(db) {
  return new DataLoader<string, Post[]>(async (authorIds) => {
    const posts = await db.query(
      'SELECT * FROM posts WHERE author_id = ANY($1)', [authorIds],
    );
    // Group by author_id so DataLoader maps 1 result per key
    return authorIds.map(id => posts.filter(p => p.author_id === id));
  });
}

// Register per-request in GraphQL context
const server = new ApolloServer({
  context: ({ req }) => ({
    user:        req.user,
    postsLoader: createPostsLoader(db),  // fresh instance = fresh cache
  }),
});

// In resolver
@ResolveField('posts', () => [Post])
getPosts(@Parent() user: User, @Context() ctx) {
  return ctx.postsLoader.load(user.id);
}
Always create a new DataLoader instance per request. Sharing across requests leaks data between users.
Subscriptions & Real-time Real-time

Subscriptions with PubSub

▶
import { PubSub } from 'graphql-subscriptions';
import { Subscription, Resolver, Mutation, Args } from '@nestjs/graphql';

const pubSub = new PubSub();

@Resolver()
export class PostsResolver {
  @Mutation(() => Post)
  async createPost(@Args('input') input: CreatePostInput) {
    const post = await this.postsService.create(input);
    pubSub.publish('POST_CREATED', { postCreated: post });
    return post;
  }

  @Subscription(() => Post, {
    // Optional filter — only push to subscribers of matching authorId
    filter: (payload, variables) =>
      payload.postCreated.authorId === variables.authorId,
  })
  postCreated(@Args('authorId') authorId: string) {
    return pubSub.asyncIterator('POST_CREATED');
  }
}
# Client subscription query
subscription {
  postCreated {
    id
    title
    author { name }
  }
}
For production, replace in-memory PubSub with graphql-redis-subscriptions (RedisPubSub) so events work across multiple server instances.
Auth & Guards in GraphQL Security

GqlAuthGuard & CurrentUser decorator

▶
// gql-auth.guard.ts — adapts HTTP guard for GraphQL context
import { ExecutionContext } from '@nestjs/common';
import { GqlExecutionContext } from '@nestjs/graphql';
import { AuthGuard } from '@nestjs/passport';

export class GqlAuthGuard extends AuthGuard('jwt') {
  getRequest(context: ExecutionContext) {
    const ctx = GqlExecutionContext.create(context);
    return ctx.getContext().req;   // extract HTTP req from GQL context
  }
}

// current-user.decorator.ts
import { createParamDecorator, ExecutionContext } from '@nestjs/common';

export const CurrentUser = createParamDecorator(
  (data: unknown, ctx: ExecutionContext) => {
    const gqlCtx = GqlExecutionContext.create(ctx);
    return gqlCtx.getContext().req.user;
  },
);

// Usage in resolver
@UseGuards(GqlAuthGuard)
@Query(() => User)
me(@CurrentUser() user: User) {
  return this.usersService.findOne(user.id);
}
GraphQL Error Handling Resilience

GraphQL errors vs HTTP errors

▶

GraphQL always returns 200 OK at HTTP level. Errors are reported inside the errors array in the response body.

// GraphQL error response shape
{
  "data": { "user": null },
  "errors": [{
    "message": "User not found",
    "locations": [{ "line": 2, "column": 3 }],
    "path": ["user"],
    "extensions": { "code": "NOT_FOUND" }
  }]
}

// Throw user-facing GraphQL errors in resolvers
import { GraphQLError } from 'graphql';

@Query(() => User)
async user(@Args('id') id: string) {
  const found = await this.usersService.findOne(id);
  if (!found) throw new GraphQLError('User not found', {
    extensions: { code: 'NOT_FOUND' },
  });
  return found;
}

// NestJS HttpException is auto-translated by ApolloDriver — no extra config
Never expose raw stack traces or internal error details in the extensions field in production. Use formatError in Apollo config to sanitize.

formatError & custom error masking

▶
// app.module.ts — ApolloDriver config
@Module({
  imports: [
    GraphQLModule.forRoot<ApolloDriverConfig>({
      driver: ApolloDriver,
      autoSchemaFile: true,
      formatError: (error) => {
        const code = error.extensions?.code;
        // Expose details only for known user-facing errors
        if (['NOT_FOUND', 'BAD_USER_INPUT', 'UNAUTHENTICATED'].includes(code)) {
          return error;
        }
        console.error(error);  // log for internal investigation
        return new GraphQLError('Internal server error');
      },
    }),
  ],
})
GraphQL Performance & Best Practices Production

Query complexity & depth limiting

▶

Without limits, clients can craft deeply nested or expensive queries that exhaust server resources.

import { createComplexityPlugin } from 'graphql-query-complexity';

// ApolloDriver plugins option
plugins: [
  createComplexityPlugin({
    maximumComplexity: 100,
    estimators: [
      fieldExtensionsEstimator(),   // reads @complexity from resolver
      simpleEstimator({ defaultComplexity: 1 }),
    ],
    onComplete: (complexity) => {
      if (complexity > 100) throw new GraphQLError('Query too complex');
    },
  }),
]

// Depth limiting
import depthLimit from 'graphql-depth-limit';
validationRules: [depthLimit(5)]   // max 5 levels of nesting
Set both complexity and depth limits in any public GraphQL API. Without them, a single crafted query can bring down the server.

Persisted queries & caching

▶
TechniqueBenefit
Persisted Queries (APQ)Client sends hash instead of full query text — reduces bandwidth; server caches parsed AST
Response cachingCache resolver results by arguments (e.g., with Redis + @cacheControl directive)
DataLoaderBatch + deduplicate DB calls per request tick
Schema introspection offDisable introspection in production to prevent schema enumeration
// Disable introspection in production
GraphQLModule.forRoot({
  introspection: process.env.NODE_ENV !== 'production',
  playground:    process.env.NODE_ENV !== 'production',
})

// @cacheControl directive on SDL (schema-first)
type User @cacheControl(maxAge: 60) {
  id:   ID!
  name: String!
}
Combine persisted queries with a CDN for public/anonymous queries. This brings GraphQL latency close to REST with proper HTTP caching semantics.
