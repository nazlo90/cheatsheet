Node.js Event Loop & I/O Core

libuv, phases & the Node event loop

▶

Node.js runs JS on V8 and delegates async I/O to libuv, which uses OS-level mechanisms (epoll, kqueue, IOCP). The event loop has six phases executed in order per tick:

PhaseWhat runs here
timerssetTimeout / setInterval callbacks whose delay has expired
pending callbacksI/O errors deferred from the previous tick
idle / prepareInternal libuv use
pollNew I/O events; blocks here if queue empty (up to timer deadline)
checksetImmediate callbacks
close callbackssocket.on('close', …)

Between every phase transition, Node drains process.nextTick queue then Promise microtasks (same as browser microtasks).

setTimeout(() => console.log('timeout'), 0);
setImmediate(() => console.log('immediate'));
process.nextTick(() => console.log('nextTick'));
Promise.resolve().then(() => console.log('promise'));
// nextTick → promise → timeout → immediate
// (timeout vs immediate order is non-deterministic outside I/O)
process.nextTick fires before Promises and can starve the event loop if called recursively. Prefer queueMicrotask() for user-land microtasks.

Non-blocking I/O patterns

▶

Node achieves concurrency without threads by offloading I/O to the OS and using callbacks/promises when work completes.

import fs from 'node:fs/promises';

// Parallel reads — don't await sequentially!
const [a, b] = await Promise.all([
  fs.readFile('a.txt', 'utf8'),
  fs.readFile('b.txt', 'utf8'),
]);

// Streaming large files instead of loading into memory
import { createReadStream } from 'node:fs';
createReadStream('huge.csv').pipe(processStream).pipe(res);
CPU-bound work (image processing, crypto, compression) blocks the event loop — offload it to a Worker thread or child process.

// ESM

// ESM __dirname equivalent
import { fileURLToPath } from 'node:url';

You cannot require() an ESM-only package from CJS. Dual-publish packages ship both formats in package.json "exports" field.

The "exports" field replaces "main" and "module". Deep imports not listed are blocked — good for encapsulation.

Streams & Buffers Performance

Stream types & back-pressure

▶

Streams process data in chunks, avoiding loading entire files into memory. Node has four stream types:

TypeExampleKey methods
Readablefs.createReadStream(), reqpipe(), read(), 'data' event
Writablefs.createWriteStream(), reswrite(), end()
DuplexTCP socketsBoth readable and writable
Transformzlib.createGzip()Modifies data passing through
import { pipeline } from 'node:stream/promises';
import { createReadStream, createWriteStream } from 'node:fs';
import { createGzip } from 'node:zlib';

// pipeline handles back-pressure and cleanup automatically
await pipeline(
  createReadStream('input.log'),
  createGzip(),
  createWriteStream('input.log.gz')
);
Back-pressure: when a writable can't keep up, write() returns false. Pause the readable until 'drain' fires. pipeline() handles this automatically — use it over raw .pipe().

Buffers & encoding

▶
// Buffer = fixed-size chunk of raw memory (uint8)
const buf = Buffer.from('hello', 'utf8');
buf.toString('hex');   // '68656c6c6f'
buf.toString('base64'); // 'aGVsbG8='

Buffer.alloc(10);       // zero-filled safe allocation
Buffer.allocUnsafe(10); // faster but may contain old data

// Convert stream chunks to string safely
const chunks = [];
stream.on('data', c => chunks.push(c));
stream.on('end', () => { const result = Buffer.concat(chunks).toString(); });
Always specify encoding when calling toString(). Default is UTF-8. Use TextDecoder for incremental decoding of multi-byte chars split across chunks.
HTTP & Express Practical

Raw http module vs Express

▶
// Raw Node HTTP
import http from 'node:http';
http.createServer((req, res) => {
  res.writeHead(200, { 'Content-Type': 'application/json' });
  res.end(JSON.stringify({ ok: true }));
}).listen(3000);

// Express equivalent
import express from 'express';
const app = express();
app.use(express.json());

app.get('/users/:id', async (req, res, next) => {
  try {
    const user = await db.findById(req.params.id);
    if (!user) return res.status(404).json({ error: 'Not found' });
    res.json(user);
  } catch(e) { next(e); }
});

// Error-handling middleware (4 args)
app.use((err, req, res, next) => {
  res.status(500).json({ error: err.message });
});
Always call next(err) in async route handlers — uncaught promise rejections won't reach the error middleware otherwise. Express 5 handles this automatically.

Middleware chain & common patterns

▶
// Middleware = (req, res, next) => void

// Auth middleware
function auth(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1];
  if (!token) return res.status(401).json({ error: 'Unauthorized' });
  req.user = verifyToken(token);
  next();
}

// Router grouping
const router = express.Router();
router.use(auth);
router.get('/', listUsers);
router.post('/', createUser);
app.use('/api/users', router);

// Rate limiting (express-rate-limit)
import rateLimit from 'express-rate-limit';
app.use(rateLimit({ windowMs: 15*60*1000, max: 100 }));
Middleware order matters. Place express.json(), CORS, and rate-limit before routes. Place the error handler last.
Cluster & Worker Threads Scalability

Cluster vs Worker Threads vs child_process

▶
APIUse caseMemoryCommunication
clusterScale HTTP servers across CPU coresSeparate processesIPC messages
worker_threadsCPU-bound JS work (crypto, parsing)Shared ArrayBufferpostMessage / SharedArrayBuffer
child_processRun external programs / shell commandsSeparate processesstdin/stdout/IPC
// Cluster — primary forks one worker per CPU core
import cluster from 'node:cluster';
import os from 'node:os';

if (cluster.isPrimary) {
  os.cpus().forEach(() => cluster.fork());
  cluster.on('exit', (w) => { console.log(`Worker ${w.process.pid} died`); cluster.fork(); });
} else {
  // each worker runs the HTTP server independently
  startServer();
}

Worker Threads for CPU-bound tasks

▶
// worker.js
import { workerData, parentPort } from 'node:worker_threads';
const result = heavyCompute(workerData);
parentPort.postMessage(result);

// main.js
import { Worker } from 'node:worker_threads';

function runWorker(data) {
  return new Promise((resolve, reject) => {
    const w = new Worker('./worker.js', { workerData: data });
    w.on('message', resolve);
    w.on('error', reject);
  });
}

// Transferable objects — zero-copy buffer transfer
const buf = new SharedArrayBuffer(1024);
w.postMessage({ buf }, [buf]); // transfer ownership, not copy
Use a worker pool library (e.g., piscina) instead of manually managing workers. Spawning workers per-request is expensive.

Security best practices

▶
  • Supply chain: pin dependencies, use npm audit in CI, consider socket.dev for behavioral analysis
  • Secrets: never commit .env; use environment variables injected at runtime
  • Prototype pollution: validate/sanitize untrusted JSON before merging into objects; use Object.create(null) for plain maps
  • Path traversal: always use path.resolve() and validate that resolved path is within expected root
  • eval / child_process: never pass unsanitized user input to eval, exec, or Function()
  • HTTPS: enforce TLS, set security headers (helmet), implement CORS allowlist
// Path traversal prevention
const root = path.resolve('./public');
const target = path.resolve(root, userInput);
if (!target.startsWith(root + path.sep)) throw new Error('Forbidden');

// helmet — set security headers in one line
import helmet from 'helmet';
app.use(helmet());
Run Node as a non-root user in containers. Set --max-old-space-size to prevent OOM from consuming all host memory.
